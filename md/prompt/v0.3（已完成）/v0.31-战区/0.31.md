# v0.31 战区系统设计

## 执行摘要

v0.31 只新增 Theater System。它位于 Region 层之上，是低频、稳定、可计算的战略分区层，用于降低后续 AI/元帅系统输入复杂度。

当前层级：

```text
Hex      战术单位移动
Region   省级控制 / 补给 / 规则
Theater  战区战略分区，v0.31 新增
```

非目标：

- 不做 FrontLine。
- 不做自动布防。
- 不做攻势方向规划。
- 不做集团军 AI 复杂决策。
- 不做经济 / 科技 / 后勤扩展。
- 不改现有 hex 战斗规则。
- 不改 Region 规则引擎。
- 不改 UI 渲染。

## 战区生成规则

v0.31 禁止动态复杂聚类、graph clustering、AI 实时生成战区结构。

初始测试地图固定生成四个战区：

- `NorthWest`
- `NorthEast`
- `SouthWest`
- `SouthEast`

划分规则：

```text
按 Region.displayHexes 的平均 q/r 与 map.width/map.height 中点比较。
q 在西半区 -> West，否则 East。
r 在北半区 -> North，否则 South。
```

生成入口：

```swift
TheaterSystem().makeInitialFixedTheaters(map: state.map, divisions: state.divisions, turn: state.turn)
```

`DataLoader.loadInitialGameState()` 在 v0.2 Region 数据成功加载后自动挂载四战区。Region 数据失败时，Theater 保持空，不影响 hex 系统。

## Region ↔ Theater 映射

规则：

- `HexCoord -> RegionId` 由 `MapState.hexToRegion` / `HexTile.regionId` 提供。
- `RegionId -> TheaterId` 由 `TheaterState.regionToTheater` 提供。
- `TheaterNode.regionIds` 反向保存战区内 Region 集合。
- Theater 不直接控制 hex。
- Theater 不直接参与战斗计算。

查询流程：

```swift
let regionId = state.map.region(for: hex)
let theaterId = state.theaterState.regionToTheater[regionId]
let theater = state.theaterState.theaters[theaterId]
```

## 战区指标

`TheaterSystem` 刷新派生字段：

- `controlRatios`：按 Region 覆盖 hex 面积计算。
- `controllingFaction`：控制面积比例最高阵营。
- `victoryPointArea`：按城市 VP × Region hex 面积聚合。
- `frontWeight`：简化值，只统计战区内 Region 与不同控制方邻居的接触数；不是 FrontLine。
- `unitIds`：当前位于该战区的单位。
- `supportEligibleUnitIds`：可行动且补给正常单位。

AI/元帅系统 v0.31 只读：

- Theater 控制比例。
- Theater 内 Region 列表。
- 简化 `threatScore`。
- 单位数量。

不读取 hex 级细节。

## 扩张逻辑

场景：阿登突破进入新 Region / 新战区。

入口：

```swift
func expandTheater(
    state: TheaterState,
    map: MapState,
    divisions: [Division],
    breakthroughRegionId: RegionId,
    faction: Faction
) -> TheaterExpansionResult
```

流程：

1. 根据 `breakthroughRegionId` 找目标战区。
2. 若 Region 尚未映射，按固定四分规则映射。
3. 临时加入目标 Theater。
4. 重算该 Theater 控制比例。
5. 若 `controlRatio[faction] >= 0.70`，标记正式 active。
6. formalized 后，该区域单位进入该 Theater unit pool。

返回：

```swift
enum TheaterTransition {
    case none
    case provisional(theaterId: TheaterId, ratio: Double)
    case formalized(theaterId: TheaterId, faction: Faction, ratio: Double)
    case retired(theaterId: TheaterId)
}
```

## 退役逻辑

入口：

```swift
func retireTheaters(
    state: TheaterState,
    map: MapState,
    divisions: [Division],
    faction: Faction
) -> TheaterState
```

触发条件：

```text
当前 active 战区控制方为 faction
且所有相邻 active 战区 controllingFaction 均为 faction
```

执行：

1. Theater 标记为 `inactive`。
2. 原 `unitIds` 平均分配到相邻 active Theater。
3. 不做权重优化。
4. 不改单位物理位置。
5. 刷新派生字段。

## 战区协同支援接口

v0.31 只提供接口，不做 AI 决策、不做战力评估、不做调度。

API：

```swift
func requestSupport(
    from theaterId: TheaterId,
    to targetTheaterId: TheaterId,
    in state: TheaterState,
    policy: SpilloverPolicy,
    reason: String
) -> TheaterSupportRequest?

func getAvailableForces(_ theaterId: TheaterId, in state: TheaterState) -> [String]

func notifyThreat(
    theaterId: TheaterId,
    sourceRegionId: RegionId?,
    threatScore: Int,
    message: String,
    in state: TheaterState
) -> TheaterState
```

`spilloverPolicy` 当前仅记录接口意图。后续 MetaAgent / 元帅 Agent 决定是否使用相邻战区兵力。

## 性能约束

必须遵守：

- 战区生成 O(n)，n = Region 数。
- 不做 graph clustering。
- 不做全局 FrontLine 重算。
- 每回合最多 1 次 theater update。
- `TheaterSystem.updateTheaters` 使用 `lastUpdatedTurn` 防重复更新。
- 扩张流程只处理变动 Region + 目标 Theater 指标。
- UI 不应在 body / draw loop 内运行 Theater 计算。

当前实现复杂度：

- 初始固定四分：O(n)。
- 重新派生指标：O(n + d)，d = Division 数。
- 支援接口：O(1) 到 O(k)，k = Theater 内单位数。
- 大图测试覆盖 30x20 Region。

## 测试模式

`TheaterSystemTests` 覆盖：

- 四战区固定生成。
- 每个 Region 正确归属 Theater。
- `hex -> region -> theater` 映射。
- 单位按当前 Region 进入 Theater unit pool。
- 扩张低于 70% 保持 provisional。
- 扩张达到 70% formalized。
- 战区退役标记 inactive。
- 支援接口返回可用兵力。
- threat notification 记录。
- AI summary 不暴露 hex 级细节。
- 30x20 大图性能 smoke test。

## 后续 v0.32 扩展点

建议 v0.32 仍基于 Theater，不直接跳到 FrontLine：

- Theater ownership event stream。
- Theater threat score 更细，但仍不读全 hex。
- Theater-level MetaAgent context。
- Theater directive 数据结构。
- Theater 支援请求仲裁。
- Theater 与 future FrontLine 的接口边界。

仍禁止在 v0.32 初期直接加入复杂自动布防 / 攻势规划，除非 Theater 指标和事件流已稳定。
