# v0.34 Tiled地图导入接口预研与需求清单

为了在 v0.35 能够直接丢入 Tiled 导出的 JSON 并在游戏中自动生成地图、省份和战区，**v0.34 需要 Codex 先铺垫好“解析器、适配器和自动拓扑算法”**。

当前项目的数据结构（`ScenarioDefinition` 和 `RegionDataSet`）是高度解耦的，但很多关系（如省份邻接 `neighbors`）是**手写硬编码**的。Tiled 输出的是“视觉格网”和“元数据（Metadata）”，因此 0.34 必须解决**从视觉网格推导战略规则**的问题。

你需要让 Codex 在 0.34 完成以下四大核心接口与模块：

## 1. Tiled JSON 原始数据映射层 (Codable Models)

Tiled 导出的 JSON 有固定的 Schema。Codex 需要写一套纯解析层，原封不动地把 Tiled JSON 读进内存。

**需要预先准备的类型：**
*   `TiledMap`: 顶层结构，解析 `width`, `height`, `hexsidelength`, `staggeraxis`, `layers`, `properties`。
*   `TiledLayer`: 图层，可以是 `tilelayer` (用于刷地形和省份色块)、`objectgroup` (用于放置城市、补给点、初始单位)。
*   `TiledProperty`: 键值对解析，你需要用这个来读取自定义的 Metadata (如 `regionId: "bastogne"`)。

## 2. 核心难点：自动拓扑生成算法 (Topology Auto-Generator)

在现有的 `ardennes_v02_regions.json` 中，省份的 `neighbors` 和 `edges` 都是配置写死的。而 Tiled 里你只能用刷子给离散的格子涂上 `regionId`。

**需要 Codex 在 0.34 实现这样一个算法模块：**
1.  **扫描邻接**: 遍历 `TiledMap` 中所有的坐标，如果 `hexA`（属于 region_X）的相邻格是 `hexB`（属于 region_Y），则自动推导出 `region_X` 与 `region_Y` 相邻。
2.  **自动连边算法**: 自动生成 `RegionNode.neighbors`。
3.  **自动生成 Edges**: 扫描交界处的图块元数据，如果在两省交界格子上检测到道路（`hasRoad=true`）或河流（`hasRiverCrossing=true`），则自动生成 `RegionEdge` 并计算移动惩罚。
4.  **行政中心推导**: 如果不专门指定，系统可以自动取省份质心所在的 hex 作为 `representativeHex`。

## 3. Tiled 到 游戏内部状态 的适配器 (TiledMapAdapter)

需要一个转换工厂接口，将解析后的 `TiledMap` 翻译成引擎需要的结构：

```swift
protocol TiledMapAdapter {
    // 将 Tiled 原生 JSON 转为现有的 Scenario 和 Region 数据结构
    func convert(tiledMap: TiledMap) throws -> (ScenarioDefinition, RegionDataSet)
}
```

**它的工作流应该是：**
1. 读取“地形图层”，初始化所有 `HexTile`。
2. 读取“省份图层”或单元格 metadata，建立 `HexCoord -> RegionId` 字典。
3. 调用第 2 步的**自动拓扑算法**，生成完整的 `RegionDataSet`。
4. 读取“对象图层 (Object Group)”，把 Tiled 里打的点转换为 `SupplySource` 和 `Objective`。

## 4. 数据加载管道扩展 (DataLoader)

在现有的 `DataLoader.swift` 中，预留并对接 Tiled 读取接口：

```swift
// DataLoader.swift
func loadTiledScenario(named resourceName: String) throws -> GameState {
    // 1. 读取 tiled json
    let tiledMap = try loadJSON(TiledMap.self, named: resourceName)
    // 2. 通过 Adapter 转换
    let (scenario, regionData) = try TiledAdapter().convert(tiledMap: tiledMap)
    // 3. 构建 GameState
    // ...
}
```
结合 v0.31 的逻辑，只要产生了正确的 `RegionDataSet`，后续的战区（Theater）、前线（FrontLine）和部署状态（WarDeployment）都可以自动顺理成章地派生出来，**不需要改这些高级系统的代码**。

## 5. Tiled 元数据(Metadata) 契约字典 (预先定死)

为了让 Codex 写代码有明确目标，你们需要在 0.34 提前确定好一套给 Tiled 用的“自定义属性/元数据”命名规范。Codex 将根据这些 Key 来编写读取逻辑：

*   **Hex 属性**: `terrain: String` (plain/forest/mountain/city), `hasRoad: Bool`, `hasRiver: Bool`
*   **Region 对象属性**: `id: String`, `owner: String`, `controller: String`, `infrastructure: Int`, `factories: Int`
*   **点对象属性**: `pointType: String` ("capital" / "supply" / "unit"), `faction: String`, `victoryPoints: Int`

---
### 总结建议

在指示 Codex 实施 v0.34 时，向他明确强调：
由于后续的高级抽象（Theater, FrontLine）依赖精确的省份拓扑，**“基于格网自动推导邻接关系（RegionGraph Auto-Generation）”** 是 v0.34 最关键、最不容有错的硬骨头，必须在 0.34 编写完善的单元测试验证这一点，否则后续的地图导入全都会崩在“找不到路径”上。
