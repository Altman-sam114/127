# v0.34 MapEditor 开发提示词 (Prompt specifically for Codex)

> **温馨提示**：在向 Codex 下发布此任务前，请建立好 `MapEditor` 目录，或者直接在 Xcode 工程内开辟一个新的 Target/Folder。然后将下方的全部文本发送给 Codex。

## 针对 Codex 的角色设定与环境要求

你是资深 Apple 原生生态工程师、地图编辑器架构师和 SpriteKit 专家。
当前我们要为一款 iOS/macOS 的六角格二战大战略游戏开发一个 **最小化可视化地图编辑器 (MapEditor)**。

**当前痛点**：项目已经进化至 v0.33，地图架构分为地块(Hex) -> 省份(Region) -> 战区(Theater) 三层结构（不一定是严格三层，而是数据颗粒度的包含关系）。现有测试文件 `ardennes_v02_regions.json` 全由硬编码写入，非常难以排错和扩展。
**核心目标**：在项目的 `MapEditor/` 目录下创建一个原生的可视化编辑器。它需要通过图形界面实现直接画图，并能直接输出能被现有游戏引擎 `DataLoader` 无缝加载的 `ScenarioDefinition` 和 `RegionDataSet` JSON 数据。禁止开发复杂的转换中间件。

要求做到绝对的零代码解耦——即在编辑器中画好地形和省份，一键 Export，丢入游戏项目的 `Data/` 目录中点 Run 即可运行 `GameState`。

## 核心功能需求清单

### 1. 基础工程与数据管理框架
*   **技术栈**：推荐使用 SwiftUI 构建界面工具栏面板，SpriteKit (或纯粹的 Canvas) 作为渲染六角网格的交互视口。
*   **文件管理系统**：
    *   **新建地图**：输入长宽 (Width x Height) 构建空白地图。
    *   **动态调整大小**：允许在已有的地图四周扩张或裁剪行列。
    *   **读取/保存**：可以将地图状态随时保存为本地存档，也可以一键从系统文件中重新加载地图。可以覆盖存档也可以删除重建。
*   **约束前提**：代码规范遵循 `WWIIHexV0` 工程中的 `Core/` 数据协议和 Swift 最新严格并发特性。

### 2. 多重交互模式 (UI 操作模式)

你不需要设计严格的显示渲染层级（比如非要三层渲染），但你的界面必须提供下方 **四个逻辑交互模式 (Modes)** 的按钮切换：

#### Mode A: 地块绘制 (Hex Painter Mode)
*   **笔刷选择**：平原 (`plain`)、城市 (`city`)、森林 (`forest`)、山地 (`mountain`)、丘陵 (`hill`)。
*   **叠加属性**：可以勾选路网 (`hasRoad`) 并点击/拖拽涂抹地形。
*   **补给网络**：可以在这层选取 Hex 设为补给站（`SupplySource`），并设定所属阵营（`Faction`）。

#### Mode B: 省份捏合 (Region Builder Mode)
*   操作逻辑：左侧栏新建一个 `Region`（可以编辑 ID，所属控制方 controller，基础设施情况）。
*   用刷子在 Hex 图块上涂抹，将被涂抹的点归于当前选中的 Region。支持刷子涂抹（添加 Hex）与橡皮擦（移除 Hex 从该 Region）。
*   视觉反馈：高亮显示当前处于同一 Region 的所有地块。

#### Mode C: 战区统筹 (Theater Assignment Mode)
*   操作逻辑：基于上一步画好的 Region，在这一层进行打组。
*   创建一个 `Theater`，例如（东部战区）。然后鼠标点选图中的 Region 色块，将它们指派到该战区下。

---

### 3. 最关键核爆点：智能推导与导出器 (Smart JSON Exporter)

游戏项目的核心验证强依赖于 `MapState` 复杂的邻接拓扑验证 (`validateRegionGraph`)。地图编辑器必须在点击“导出”时，在后台利用数学算法跑通以下推演：

1.  **自动计算 Region Neighbors（省份邻居）**：
    遍历全图 Hex，如果 `Region A` 的某个格子和属于 `Region B` 的某个格子**在六维坐标 (q,r) 上直接相邻**，自动判定 A 和 B 互相为 Neighbor。
2.  **自动生成 Region Edges（省份边缘与移动代价）**：
    检测邻近的两个 Region 边界（沿线格子），如果这两个交界格子上都有 `hasRoad=true`，则自动生成一条拥有 `hasRoad=true` 属性的 `RegionEdge` 连线节点！
3.  **自动计算 Administrative Center (行政中心)**：
    自动获取 Region 中所有点构成的面积重心所在的 Hex，指定为 `representativeHex`。
4.  **符合规范序列化**：
    按严格的数据类型序列化成为两个文件：一个是存放地形与补给点 `ScenarioDefinition` 的 JSON；另一个是符合 `RegionDataSet` schema，包含 HexToRegion 映射，省份连线边集，控制方等等的 JSON 文件。

### 4. 落地与验证开发指引

在编写这个模块时，你 (Codex) 必须遵循以下阶段：

1.  **准备核心数学算子**：参考业界经典的奇偶行偏移（或使用立方体坐标 `x, y, z` 转 `q, r` ）算法，彻底搞定坐标系换算和屏幕鼠标到六角网格的“点选取 (Hit-Test)”碰撞判定。
2.  **构建核心 ViewModel**：写出编辑器管理整个地图生命周期的全量数据池结构。保持撤销/重做或更新的简洁性。
3.  **UI 开发**：先做出能跑起来的最简丑陋可用版。用 SwiftUI 放一堆 Button 选模式，右侧直接嵌入 `SKView`。
4.  **连通性测试 (Integration Testing)**：
    创建一个名为 `MapEditorOutputTests.swift` 的文件放到我们原有的 `Tests` 目录下。
    生成一张小的新建测试图（例如测试两片平原接壤然后划分战区），导出 JSON，然后直接通过现有的 `DataLoader` 加载该数据，如果 `validateRegionGraph()` 没有报错，并且成功返回了完整的 `GameState`（能成功构建 FrontLine 并且没有抛错），即代表你的任务全面胜利。

**附录**：
地形种类、阵营枚举、核心战区类都可以直接从主 App 代码 `WWIIHexV0/Core/` 库里面 `import` 引用，不要自己在编辑器项目里复制并重新建模（违反 DRY 原则）。如果需要，把 `WWIIHexV0` 对应的结构放到一个公共 Target 或者直接包含其文件。

请开始你的详细架构实现。直接编写或生成该编辑器的工程文件结构，从数据管理模块开始你的编码。
