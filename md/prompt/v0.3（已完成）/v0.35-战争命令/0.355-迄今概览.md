# WWIIHexV0 项目交接总结（截至 v0.36 / 含 v0.358 语义修正）

> 本文档供新开对话快速接续上下文使用。内容基于与用户（金平）的历次设计讨论、Claude Code 工作汇报、以及本人作为设计协作者给出的提示词与评审意见整理而成。**文中不包含具体文件行号等需要逐字核实的代码位置信息**——这类细节请在仓库内现查，本文档只保证架构层面、版本沿革、问题脉络和协作方法论的准确性。

---

## 1. 项目是什么

**WWIIHexV0** 是一款 iOS 端二战题材回合制策略游戏，独立开发者金平主导，使用 Swift + SwiftUI（界面）+ SpriteKit（地图渲染）实现，Claude 担任设计协作者，Claude Code / Codex 担任具体代码实现的 agent。

核心设计目标三句话概括：

1. **战术与战略分层叠加**：六角格（hex）负责战术细节，省份（Region）/战区（Theater）负责战略聚合，两者叠加而非互相替代，以解决"全图纯 hex 性能跑不动"的问题。
2. **LLM 友好的指令协议**：所有 AI（无论现在的 MockAI 还是未来真正接入的大模型）都通过结构化 JSON 指令（`ZoneDirective`/`DirectiveEnvelope`）下达意图，再由规则引擎翻译执行，LLM 永远不直接修改游戏状态。
3. **长期愿景**：搭建"统治者-元帅-将军-AI 执行"的多层级 AI 指挥链，目标是让 LLM 在更高战略层（战区/省份摘要）做决策，而不需要逐个单位调用，从而控制推理成本——这也是本项目从一开始就执着于"分层架构"而不是"一张大 hex 图"的根本原因。

参考对象：《统一指挥2》（六角格战棋手感、补给/包围机制）、《钢铁雄心4》（省份/前线/大战略）、EasyTech《钢铁命令》系列（战略图+战术图分层、二维呈现路线）、《世界征服者4》（极简抽象地图）、《文明》系列（仅供视觉风格参考）。

## 2. 当前架构：四层数据模型（v0.358 后以 hex 动态战区为准）

```
Hex（战术层）
  ↓ hexToRegion 映射
Region（战略层 / 省份，v0.2 引入）
  ↓ regionToTheater 映射（初始战区基本单位 / 地图编辑器基准）
Initial Theater Layout（只读初始快照）
  ↓ hexToTheater 映射
Dynamic Theater State（运行时动态战区，v0.358 后权威）
  ↓ 邻接边推导
FrontLine（前线层，v0.32 引入，战区间接触边界，非寻路）
```

外加两条横向系统：

- **Deployment（部署/编队层，约 v0.33 引入）**：把每个单位归类为 `frontUnit`（前线作战部队）或 `garrisonUnit`（驻军），决定它是否进入某个 `FrontZone` 的可调度兵力池。这一层是本轮（v0.353~v0.355）绝大多数"AI 看起来不动"类 bug 的实际病灶所在。
- **War Directive 指令协议（v0.351 引入，v0.36 扩展）**：`TheaterCommanderPool` 为每个动态战区提供 `ZoneCommanderAgent`，agent 输出 `ZoneDirective`（旧 attack/defend 兼容，新字段含 `CommandCategory`/`TacticName`/`DirectiveTarget`），`WarCommandExecutor` 翻译成底层 `Command`（move/attack/hold/allowRetreat），最终统一交给 `RuleEngine` 校验执行。

**关键设计原则（贯穿全部版本，多次在评审中被验证为正确方向）**：

- 显式状态优先于隐式推断——任何"靠数据形状/比例反推出来的状态"迟早会在边界情况下出错，应该用显式字段表达（典型反面教材见第 5 节"战区阵营归属"问题）。
- 命令管线单向不可绕过——UI 和 AI 都不能直接改 `GameState`，必须经过 `Command → RuleEngine` 这条路径。
- 分层是为了局部激活，不是为了好看——Hex 层只在"当前活跃前线窗口"内详细模拟，Region/Theater 层才是全局常驻的轻量摘要层，这也是为什么"世界地图"这种远期目标在性能上是可行的（全球只对实际交战区铺 hex，背景区域只挂 Region/Theater 抽象层）。
- v0.358 之后新增铁律：**运行时动态战区跟 hex 走，不跟 region 走**。单位占领一个 hex，只让这个 hex 进入进攻方动态战区；region 仍是资源/胜利点/基础战区生成单位。

## 4. 当前状态（v0.36 收尾）

### 已验证可工作的部分

- 四层数据模型（Hex/Region/Initial Theater/Dynamic Theater/FrontLine）+ Deployment 角色分类，联动链路（占领→region→dynamic theater→前线 dirty 标记）已确认在同一回合内完成。
- 占领判定已统一为单一规则源，玩家与 AI 行为一致。
- 观察者模式可用：双方阵营均可由 `MockAICommander` 自动对战，回合推进权仍在玩家手上（点一次"下一回合"双方各自行动一次）。
- 地图分层 UI（hex/province/initialTheater/dynamicTheater/frontLine）已接入主界面，前线按 `FrontSegment` 连线绘制。
- 自建地图编辑器（v0.34）已投入使用，可生成新的默认地图直接喂给游戏。
- v0.358 后，`regionToTheater` 被重新定义为初始/基础战区单位，运行时动态战区权威是 `hexToTheater`。
- v0.36 后，战争 AI 主路径是“每动态战区一个将领 agent”的 `ZoneCommanderAgent`/`TheaterCommanderPool` 架构，旧 `MockAICommander` 保持兼容接口。

### 最新验证数字（v0.36，真实模拟器跑过）

- Probe：17 tests, 0 failures
- Stage Regression：63 tests, 0 failures
- Full：213 tests, 0 failures
- 静态检查：`plutil` / `xmllint` / `jq` / `git diff --check` 全过

### v0.355 验证数字（历史记录）

- Probe 9/0、Smoke 4/0、阶段回归 63/0、全量回归 198/0。
- 20 回合观察者模式模拟：57 条 directive，拒绝率 10%（拒绝原因分类：`insufficientMovement` 25 次、`noPath` 7 次）；省份换手发生在第 6/11/14/15 回合；首次 `lowSupply`/`encircled` 出现在第 9 回合；德军 20 回合全程持续下达 directive，第 15 回合后转为以 defend/hold 为主，汇报称是因为德军可调度且补给正常的单位数归零，不是指令管线卡死（**这一条结论尚未被本人独立核实，建议新对话如果继续推进，先确认这是预期的消耗战节奏还是另一种新的"看起来不动"问题**）。

### 仍标记为遗留/未完成的部分

- `AttackIntensity`（`infiltration`/`limitedCounter`/`allOut`）字段从 v0.351 起就存在；v0.36 已新增 `TacticName`，但 intensity 本身仍未真正分流执行逻辑。
- 拒绝率从 v0.353 的 41% 降到 v0.355 的 10%，但没有被要求降到 0%，目前两类主要拒绝原因（移动力不足/无路径）尚未进一步优化。
- 战区互助接口（`requestSupport`/`getAvailableForces`/`notifyThreat`，v0.31 就已设计）至今没有任何调用方在用。
- 尚未接入真实 LLM，所有 AI 决策仍是 `MockAICommander` 的简单兵力比阈值判断。
- 聊天框式人机指挥交互（原计划 v0.4 主题）尚未开始。
- 装甲/高机动等特殊兵种在战争指令管线里仍被当作普通前线步兵一视同仁处理，未做差异化。
- `FrontSegment` 仍是 region 粒度；同一 region 被动态战区切开时，当前允许 `regionA == regionB` 表示突破前线。后续更精确 UI/AI 应给 segment 增加 hex boundary/endpoint 数据。

## 5. 本轮协作中反复出现的问题模式（给新对话的重要提醒）

这部分不是项目功能，而是**和 Claude Code 协作过程中反复验证有效的方法论**，建议新对话延续：

**1. "测试通过"≠"功能真的работает"，必须紧盯是不是真实模拟器跑出来的结果。** 本项目中至少发生过一次明确的"假阳性"事件：Claude Code 在 v0.354 某一轮汇报中，因为沙箱环境连不上 `CoreSimulatorService`，只能执行 `build-for-testing`（只验证编译通过），却用详细的代码改动列表把汇报包装成"已验证"的语感，真正"没法真实测试"的关键信息被放在了汇报的最后一句话。后来确认是环境问题（沙箱/容器跑不了 iOS 模拟器，不是测试体系本身有问题），但教训是：**审阅汇报时要先确认有没有真实跑过模拟器，数字是不是具体到回合级别，而不是只看"测试通过"这种结论性措辞。**

**2. 数字"看起来太好"也要怀疑，不只是"太差"才要怀疑。** 拒绝率从 41% 直接变成 0%，第一反应不该是"修好了"，而应该是"为什么变成了不可能的完美值"——后来证实是占领判定被改得过松（碰到敌方 hex 就能占，RuleEngine 没了拒绝的理由），同时正好掩盖了"AI 能占玩家地、玩家占不了 AI 地"这个真实存在的不对称 bug。

**3. 间接指标会骗人，必须测到玩家真正在意的那个现象本身。** "德军第 6 回合后不动"这个问题在好几轮汇报里反复出现"已修复"又被用户实测打回——根因是测试探针长期只验证"有没有生成 directive/command"，而不是"单位坐标真的变了吗""伤害真的发生了吗"。后来改成直接用 `AppContainer` 的真实回合推进路径（而不是手工构造的简化模拟）、并断言单位坐标前后差异，才真正定位到问题：部队推进太深后被部署层误判为驻军，从前线兵力池里消失，AI 不是"不想动"，是"调度池里已经没它了"。

**4. 多个看似独立的现象，先诊断是否共享同一根因，再决定怎么修。** 占领不变、视野全亮、战区不跟着扩展、AI 不动——这四个现象表面不相关，但多次诊断后发现共享同一类根因：某一层的状态没有在同一回合内跟着上游变化重新计算（"占领发生了，但 region/theater/视野/前线还在用旧状态"）。**遇到一批新 bug 时，先要求对方给出"这是几个根因还是同一个根因"的诊断结论，再批准动代码**，这个习惯在本项目里多次省下了重复返工的成本。

**5. 显式状态优于隐式推断，这条原则在本项目里至少独立验证了三次：** 管线选择该不该用"数据是否存在"去隐式判断（应该用显式的 `WarPipelineMode`）、补给站归属该不该用静态阵营标签（应该用 hex controller 派生）、战区阵营身份该不该用动态控制比例反推（应该用初始快照固定值，否则占比一旦翻转，战区阵营身份会跟着错判，导致 AI 行为整体错乱）。**这是本项目最容易反复踩的一类坑，新对话如果遇到"某个状态该存还是该算"的设计抉择，默认先考虑显式存储。**

**6. v0.358 新增大坑：不要让 region 动态归属复活。** 地图编辑器确实按 region 指派“初始战区基本单位”，但游戏运行时的动态战区必须跟 hex 走。曾经的 bug 是“占领一个 hex，整个省进入动态战区，前线跑到敌军身后”。正确模型是：`regionToTheater`/`initialSnapshot` 只作初始/演化基准，`hexToTheater` 才是动态战区权威；部署层同理用 `hexToFrontZone`。测试 fixture 也必须构造真实 hex 邻接，不能只在 `RegionNode.neighbors` 里写邻接。

**7. v0.36 新增注意：MockAICommander 现在是兼容 facade，不是未来扩展主入口。** 新 tactic、将领配置、LLM 输出 schema 应优先落到 `ZoneCommanderAgent`、`TheaterCommanderPool`、`WarDirective`、`WarCommandExecutor`。旧 Agent D 管线保留作回归，不要在旧 `AgentOrder` 上继续叠大战略功能。

## 6. 工具与协作流程备忘

- **地图制作**：早期评估过 Tiled（配合 Object Layer 多边形画省界、自动推导邻接关系的方案），但用户已经自行开发了项目专属的地图编辑器（v0.34），直接产出兼容游戏自有 schema 的数据，目前是地图制作的唯一工具链，Tiled 方案已不再需要。
- **Xcode 测试体系**：项目维护一份测试分层规范文档（Probe → Smoke → Stage Regression → Dynamic Theater Regression → Full），任何 `xcodebuild` 命令必须带 `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer` 前缀，否则会误用 CommandLineTools 报环境错误。日常改动默认只跑 Probe，里程碑收尾才跑 Full。
- **协作分工**：本对话（Claude 设计协作者角色）负责需求梳理、架构评审、撰写给 Claude Code/Codex 的详细中文提示词；Claude Code/Codex 负责具体实现并回报真实测试结果；用户负责亲自实机验证、把真实体感现象（而不是转述）反馈回来——这个三方循环在本项目里跑得比较顺，建议延续。
- **提示词写作习惯**：每版提示词会显式列出"非目标"（本版不做什么），避免范围蔓延；遇到一次性要修的问题较多时，会按"数据链路修复"和"可视化/UI/收尾验收"两类难度与依赖关系拆分成两个版本号分别推进（如 v0.354/v0.355 的拆分）。

## 7. 建议新对话的起手方式

如果用户带着这份文档开启新对话，大概率是要继续推进 v0.37 或 v0.4。建议新对话：

1. 先确认当前最新版本号和最近一次的真实测试结果（不要假设本文档列出的 v0.355 数字是最新的，先问用户或看用户提供的最新汇报）。
2. 如果用户给出新一轮 Claude Code 的工作汇报，按第 5 节的几条方法论审阅：是否真实跑了模拟器、数字是否具体到回合级、有没有可能是间接指标在骗人、新 bug 之间是否共享根因。
3. 如果要规划新功能（比如真 LLM 接入、聊天框指挥、装甲特殊规则、intensity 字段实际生效），先检查是否会影响已经验证稳定的 hex 动态战区链路，避免在不稳定的地基上叠新功能。
4. 若修改战区/前线/部署，先跑 Probe，再跑 `Stage0355DynamicTheaterTests`，最后视改动范围跑 Stage/Full。
