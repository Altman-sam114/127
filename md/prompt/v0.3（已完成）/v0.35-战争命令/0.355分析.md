### ⚠️ 概览中存在描述偏差或需要注意的部分

#### 1. 概览第 3 节版本表中 v0.34 说法："放弃了引入 Tiled 的方案"

这是**准确的**：`MapEditor/` 目录存在，说明自建编辑器已在用。但"v0.34"这个版本号与文档自身在第6节又提到"自建地图编辑器（v0.34）已投入使用"前后一致。  
**无问题**，只是稍微需注意这个版本划分是文档历史叙述，并非代码中有 v0.34 标签。

#### 2. v0.352 描述中"`WarDirectiveRecord`"已实现

**概览原文（第59行）：**
> WarDirectiveRecord

**核查结果：**
✅ `Core/WarDirectiveRecord.swift` 文件存在（979字节）。**无问题**。

**核查结论：**
这些数字属于"某次历史测试结果"，文档自身已注明"最近一次完整汇报，已知是真实模拟器跑出来的结果"。**目前无法通过静态代码核实**，且概览第3节第7条方法论也已明确提示须先确认是否真实模拟器产出。属于**已知不可核实项**，不是幻觉，而是带免责声明的引用数据。

#### 4. 概览中提及"战区互助接口（`requestSupport`/`getAvailableForces`/`notifyThreat`，v0.31 就已设计）至今没有任何调用方在用"

- 这些**接口设计存在于数据结构层**（`TheaterSupportRequest`），但**方法调用层的函数名与概览描述不完全吻合**。
- 概览说这些接口"至今没有任何调用方在用"——**这是准确的结论**，但方法名描述稍有不精确（实际数据结构名略有差别）。整体结论没有问题。

#### 5. 【轻微幻觉】概览关于 Agent D 管线的描述

**概览原文（第52行）：**

**核查结果：**
这些类型全部存在，但目前在 v0.352 之后，**所有这些文件均被标记为 DEPRECATED**（`// DEPRECATED as of v0.352 - kept for regression reference, not invoked by default`）。README 中将 Agent D 列为"✅ 已完成"并做了完整描述，这与当前代码状态（标记废弃、不默认启用）有一定落差。

**结论**：概览在版本演进表中描述 Agent D 属实，但没有明确提及"这套管线在 v0.352 后已被标记为 DEPRECATED"——这点对新对话容易产生误导，认为 Agent D 管线仍处于活跃使用状态。这是**轻微遗漏/可能导致误解的表述**。

## 二、原 v0.3 计划（6个Agent）执行情况分析

### 背景说明

原 v0.3 计划是一套以**前线为中心、基于 Region/Province 抽象层**的系统，包含6个Agent。但由于项目后来大改为**战区制（Theater System）**，v0.3 的实际执行走上了不同路径。

---

### Agent 1：前线规则与数据模型

**原计划要求：**
- `FrontLine`, `FrontSegment`, `FrontEdge`, `ArmyGroup`, `FrontAssignment`, `FrontRiskLevel`, `FrontGapWarning`
- `RegionRef/ProvinceRef` 抽象引用
- `FrontLineDetector`, `FrontAnalyzer`, `RegionGraphProviding`, `UnitPositionProviding`
- `HexRegionAdapter`

**实际执行情况：**
- ✅ `FrontLine.swift`, `FrontSegment.swift` 存在
- ✅ `FrontLineTypes.swift` 存在（内含类型枚举）
- ✅ `FrontLineState.swift`, `FrontZone.swift`, `FrontZoneId.swift`, `FrontZoneSegment.swift` 存在（概念略有变体，用 `FrontZone` 代替 `ArmyGroup/FrontAssignment`）
- ✅ `Rules/FrontLineManager.swift` 存在（21134字节，实质上承担 FrontLineDetector + FrontAnalyzer 职责）
- ⚠️ `RegionGraphProviding` 协议：未以原计划命名存在，但 `Region.swift`（11544字节）+ `RegionMovementRules.swift` 承担了类似职责
- ⚠️ `HexRegionAdapter`：未找到此命名，因项目已全面转向 Region 层，不再需要 "hex adapter" 兼容层
- ❌ `FrontRiskLevel`, `FrontGapWarning`：以原计划命名未找到（可能被整合进其他类型或舍弃）

**结论：Agent 1 的骨架已以变体形式落地（约70%），命名和抽象层级有较大调整。**

---

### Agent 2：自动布防与调兵算法

**原计划要求：**
- `FrontDefensePlan`, `FrontPosture`, `DefenseSlot`, `UnitAssignment`, `ReserveGroup`, `ReservePurpose`
- `FrontDefensePlanner`（输出 proposedCommands，不直接改 GameState）

**实际执行情况：**
- ❌ `FrontDefensePlan`, `FrontDefensePlanner`, `DefenseSlot`, `UnitAssignment`, `ReserveGroup` 均未找到
- ✅ `WarDeploymentManager.swift`（21623字节）承担了"部署/编队"职责，实现了 `frontUnit`/`garrisonUnit` 角色分类
- ✅ `WarDeploymentState.swift`, `WarDeploymentTypes.swift` 对应部分设计理念
- ⚠️ 但原计划的"评分算法分配防区、优先填补缺口、装甲保留为预备队"这类**布防规划逻辑**，在代码中**未以 Planner 形式存在**。现有 `WarDeploymentManager` 更多是维护单元部署状态，而不是生成布防计划

**结论：Agent 2 的"部署状态管理"部分以变体落地，但"布防规划算法（生成 FrontDefensePlan）"这一核心功能实质上未实现（~20% 落地）。**

---

### Agent 3：攻势箭头与突破计划

**原计划要求：**
- `OffensivePlan`, `OffensivePlanType`（holdLine/limitedAttack/breakthrough）
- `OffensiveAxis`, `BreakthroughObjective`
- `OffensivePlanner`（BFS/Dijkstra路径、目标评分、单位筛选）

**实际执行情况：**
- ❌ `OffensivePlan`, `OffensivePlanner`, `OffensiveAxis`, `BreakthroughObjective` 均未找到
- ✅ `MockAICommander.swift`（8938字节）承担了类似"AI 攻势决策"职责，但其逻辑是简单兵力比阈值判断（ZoneDirective attack/defend），而非完整的突破轴线规划
- `WarCommandExecutor.swift`（22858字节）负责把 ZoneDirective 翻译为底层命令，但这是"执行层"，不是"计划生成层"

**结论：Agent 3 完全未实现（0%落地）。原计划的攻势箭头、突破轴线计算功能在当前代码中不存在，被简化为 MockAICommander 的阈值判断所替代。**

---

### Agent 4：元帅/将领 Agent 接口

**原计划要求：**
- `MarshalAgent`, `MarshalCommandStyle`, `TheaterAgentContext`
- `MarshalDecisionEnvelope`, `FrontDirective`
- `MockMarshalDecisionProvider`, `MarshalAgentController`
- 前线级 JSON 指令 → 调用 FrontDefensePlanner/OffensivePlanner

**实际执行情况：**
- ❌ `MarshalAgent`, `MarshalDecisionEnvelope`, `MarshalAgentController` 均未找到
- ✅ `MockAICommander.swift` 承担了类似"AI 战略决策者"的角色，输出 `DirectiveEnvelope`（包含 `ZoneDirective`）
- ✅ `WarDirective.swift` 中的 `DirectiveEnvelope`/`ZoneDirective` 是原计划 `MarshalDecisionEnvelope`/`FrontDirective` 的精简对应物
- ⚠️ 整体概念"元帅读取战区摘要 → 输出前线级计划 → 规划器拆解"的链路，实际变成了"MockAICommander 读取 FrontZone 状态 → 输出 ZoneDirective → WarCommandExecutor 拆解"，职责被压缩合并

**结论：Agent 4 的核心概念以高度精简的变体落地（约40%），但"多层级指挥链"（Marshal → General → 执行层）的设计未实现，仍是扁平单层 MockAICommander。**

## 四、结论

### 0.355概览幻觉核查
**整体无重大幻觉**，架构层面叙述与代码吻合。唯一需注意的是：
- Agent D 管线（`DecisionProvider`, `MockAIClient` 等）已在 v0.352 标记 DEPRECATED，概览没有明确提及这一点，容易被新对话误读为活跃管线。

### 原 v0.3 计划执行分析
原 v0.3 计划是基于**Region 层前线**的复杂系统（6个Agent），**目前已执行的核心精神是 Agent 1 的前线识别、Agent 4 的 AI 指令接口、Agent 6 的测试体系**，整体架构上嫁接到了 Theater（战区）制，而非原计划的 ArmyGroup/FrontAssignment 制。

**显著未执行的部分：**
1. **Agent 2 布防规划算法**（FrontDefensePlanner 分配防区、评分、生成 proposedCommands）
2. **Agent 3 攻势箭头/突破计划**（OffensivePlanner/OffensiveAxis/路径计算）
3. **Agent 5 玩家前线操作 UI**（选集团军→分配前线→设置攻势）
4. **多层级指挥链**（Marshal → General → 执行的分层 Agent 结构）
