# v0.37 审计与整改记录

## 目标

v0.37 不新增玩法机制，只做命令层地基工程：

- 默认战争 AI 路径收口为 `TheaterCommanderPool -> ZoneCommanderAgent -> ZoneDirective -> WarCommandExecutor -> RuleEngine -> WarDirectiveRecord`。
- Legacy Agent D 保留作回归参考，但默认运行路径不触发。
- 确认 v0.4 所需能力：不经过具体 `ZoneCommanderAgent` 实例，手写合法 `ZoneDirective` 也能由 `WarCommandExecutor` 执行。

## 阶段 A：只读审计结论

1. 默认战争 AI 入口是 `AppContainer.runAIIfNeeded()` -> `runAISequence()` -> `TurnManager.runAITurn(..., pipelineMode: warPipelineMode)`。`AppContainer` 默认 `warPipelineMode = .zoneDirective`，默认注入 `TheaterCommanderPool`。

2. `TurnManager.runGermanAITurn(state:)` 只转调 `runAITurn(... faction: .germany, pipelineMode: .zoneDirective)`。管线选择由 `WarPipelineMode` 显式 `switch`，不是靠 front zone 数据形状隐式判断。

3. Legacy Agent D 的真实执行只在 `.legacyAgentOrder` 分支。`AgentContextBuilder`、`AgentDecisionParser`、`AgentCommandMapper` 仍作为 `TurnManager` 兼容属性和测试对象存在；测试 target 仍覆盖 legacy 行为，这是回归用途。

4. 审计发现 `MockAICommander` 不是完全纯 facade：它内部委托 `ZoneCommanderAgent`，但 `TurnManager` 仍保留 `directiveCommander` fallback，部分测试也直接 new `MockAICommander` 当入口。默认 `AppContainer` 不注入它，但该 fallback 容易让默认路径语义变模糊。

5. `WarCommandExecutor` 对外入口是：

   ```swift
   func execute(_ directive: ZoneDirective, in state: GameState) -> WarCommandExecutionResult
   ```

   它不要求传入 `ZoneCommanderAgent` 或 agent 实例；只依赖合法 directive、GameState 和 command handler。

6. `WarCommandExecutor` 的必要上下文假设是游戏规则层假设，不是 AI 生成者假设：`directive.zoneId` 必须存在于 `state.warDeploymentState.frontZones`，attack 的 `targetTheaterId` 会映射到目标 `FrontZoneId`。

7. `ZoneDirective` 当前 Codable 字段：

   - `zoneId`
   - `type`
   - `parameters`
   - `category`
   - `tactic`
   - `commandTarget`

   单条 directive 没有 issuer 字段。

8. `DirectiveEnvelope` 当前 Codable 字段：

   - `schemaVersion`
   - `issuerId`
   - `turn`
   - `directives`
   - `commanderAgentId`
   - `theaterContext`

   来源信息在 envelope 级，不在单条 directive 级。

9. `WarDirectiveRecord` 字段包括 `issuerId`、`turn`、`faction`、`zoneId`、`directiveType`、`targetRegionIds`、`commandResults`、`diagnostics`、`category`、`tactic`、`commanderAgentId`、`commandTarget`。默认 AI 指令路径会给每条 directive 留 record；直接手工调用 `WarCommandExecutor.execute(...)` 不会自动落 `WarDirectiveRecord`，记录职责仍在 `TurnManager.runDirectiveTurn` 外层。

10. `CommandCategory` 当前只有 `offense / defense`。`TacticName` 当前只有 `standardAttack / holdPosition`。`AttackIntensity` 仍存在 `infiltration / limitedCounter / allOut`，但执行层没有按 intensity 分流。

## 阶段 B：实际整改

### 命令层收口

- 移除 `TurnManager` 的 `directiveCommander: MockAICommander?` 属性、初始化参数和 `.zoneDirective` fallback 分支。
- `.zoneDirective` 分支现在只按以下顺序产生 envelope：
  1. front zone 缺失 -> 空 `DirectiveEnvelope` + 诊断，且不 fallback legacy。
  2. 显式 `commanderPool`。
  3. `TheaterCommanderPool.automatic(for:)`。
- 测试中原本显式传入 `directiveCommander: MockAICommander()` 的 `TurnManager` 构造改为 `TheaterCommanderPool.automatic(for:)` 或默认 automatic fallback。
- `MockAICommander` 类保留，用于兼容旧测试和简单阈值行为验证，但不再是 `TurnManager` 的备用执行入口。

### 0.4 接口预留

- 未改 `WarCommandExecutor` 签名，因为审计确认它已经 issuer-agnostic。
- 新增 `CommandSystemTests.testV037WarCommandExecutorExecutesManualDirectiveWithoutIssuer`：手工构造 `ZoneDirective`，不经过 `ZoneCommanderAgent`，直接调用 `WarCommandExecutor.execute(...)`，断言能产生并执行底层 command。
- 新增 Probe：`testProbeV037ManualDirectiveExecutesWithoutCommanderIssuer`，给后续 v0.4 玩家 UI 接入提供高速回归基线。

### 测试修正

- `testMockAICommanderUsesNamedAttackThresholdBoundary` 原先用单位数量近似 1.1 / 1.3 阈值。当前分类器实际使用 `strength + attack` 对 `strength + defense`，因此改为显式设置单个单位 strength，使测试语义和真实战力比一致。

## 推迟到 1.x 的决策

撤退命令、突破/闪电战、装甲差异化、`AttackIntensity` 实际分流均推迟到 1.x。

原因：这些机制不是简单新增一个 enum case，而是共享“多回合追踪行动”骨架。它们需要解决目标位置选择、部队排布、行动完成判定、完成后防线重新形成等关联问题。过早实现任意一个都会增加后续推倒重写风险。

## 当前结论

- 默认战争 AI 路径已收口为 `TheaterCommanderPool -> ZoneCommanderAgent -> ZoneDirective -> WarCommandExecutor -> RuleEngine -> WarDirectiveRecord`。
- Legacy Agent D 只在显式 `.legacyAgentOrder` 或测试回归中使用。
- `WarCommandExecutor` 已确认支持 issuer-agnostic 执行，满足 v0.4 玩家 UI 后端接入前提。
- `AttackIntensity` 仍未分流执行，这是有意保留的 1.x 事项，不属于 v0.37 缺口。
