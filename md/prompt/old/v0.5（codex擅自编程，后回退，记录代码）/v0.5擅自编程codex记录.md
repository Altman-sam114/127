v0.5 国家与部长 Agent 执行方案
Summary
v0.5 的目标不是做完整经济、外交、科技和政变系统，而是建立“国家级 agent + 部长 agent + 指挥链仲裁”的骨架。核心交付是：高层 agent 能读取摘要、提出战略建议或目标，系统能把这些建议记录、展示、下发给前线/将领层，但不允许它们直接移动单位或绕过规则。
建议拆成 6 个 agent 并行推进，其中 A/B 先行，C/D/E/F 在接口稳定后接入。
Agent 分工
Agent A：多 Agent 架构与数据模型
负责定义 v0.5 的核心抽象。
工作内容：
扩展 GeneralAgent 为更通用的 GameAgent 或新增 PoliticalAgent / StaffAgent。
定义 agent 角色：统治者、元帅、军工部长、陆军部长、外交部长、情报部长、将领。
定义 AgentRole、AgentAuthorityLevel、AgentPersonality、AgentRelationship。
定义 agent 权限边界：高层 agent 只能输出战略意图和建议，不能直接下达单位级命令。
设计 AgentMemory 的最小结构：短期战况记忆、长期人物倾向、最近指令。
扩展 JSON 配置，例如 agents.json 或 national_cabinet.json。
交付物：
多 agent 数据结构。
示例德国国家内阁配置。
示例盟军国家内阁配置。
基础单元测试，验证 agent 配置可加载、角色权限正确。
依赖：
需要沿用 v0 的 DecisionProvider、AgentContext 思路。
不依赖 UI。
Agent B：国家级摘要与上下文构建
负责把游戏状态压缩成不同层级 agent 可读的摘要。
工作内容：
定义 NationalContext：国家总体战况、资源概况、前线摘要、威胁、目标。
定义 MinisterContext：按部门裁剪信息。
定义 TheaterContext：战区级摘要，供元帅/战区司令使用。
定义 CommanderContext：沿用或扩展将领局部战场摘要。
实现上下文构建器，确保高层 agent 不读取不该知道的全量细节。
为本地 LLM 控制 token 量，做摘要长度限制。
交付物：
AgentContextBuilder 分层版本。
国家级、部长级、战区级摘要结构。
测试用固定 GameState，能生成稳定摘要。
单元测试：不同角色拿到不同粒度信息。
依赖：
Agent A 的角色模型。
v0/v0.3 的 GameState、前线、经济占位数据。
Agent C：战略意图与仲裁系统
负责解决多个 agent 同时提出建议时，谁说了算、如何落地。
工作内容：
定义 StrategicDirective：例如进攻方向、防守重点、生产优先级、科技建议、外交建议。
定义 DirectivePriority：统治者 > 元帅 > 部长 > 将领，或按领域仲裁。
定义 DirectiveStatus：proposed、approved、rejected、active、expired、completed。
实现 DirectiveBoard 或 AgentCouncil，集中管理所有高层建议。
实现冲突检测：例如陆军部长要求补员，统治者要求立即进攻。
将高层 directive 转换为下级 agent 的 PlayerDirective 或 TheaterDirective。
交付物：
战略指令模型。
仲裁器。
冲突日志。
测试：多个 agent 输出冲突建议时，系统能稳定选出结果并记录原因。
依赖：
Agent A 的角色权限。
Agent B 的摘要。
不直接依赖 SpriteKit UI。
Agent D：部长 Agent 原型
负责第一批部长 agent 的行为模板和 Mock 决策。
工作内容：
实现军工部长：根据装备、工厂、损耗提出生产建议。
实现陆军部长：根据兵力、组织度、损失提出补员、训练、任免建议。
实现外交部长：先只输出外交态势和风险建议，不做真实外交规则。
实现情报部长：根据可见敌军、前线压力、未知区域输出威胁评估。
每个部长先用 MockAI 规则生成结构化建议，后续再接本地 LLM。
定义每个部长的 prompt 模板，但 v0.5 默认不强制调用真实 LLM。
交付物：
4 个部长 Mock provider。
每个部长的结构化 JSON 输出 schema。
每回合部长会议建议列表。
测试：固定战况下部长输出稳定、可解析、可被仲裁器接收。
依赖：
Agent B 的部长上下文。
Agent C 的战略指令模型。
Agent E：统治者与玩家角色入口
负责玩家作为统治者的交互入口，以及统治者 agent 的总体目标输出。
工作内容：
设计 RulerAgent：国家目标、政治风格、战争优先级、对将领信任。
支持玩家以统治者身份发布国家级命令。
将玩家自然语言命令转换为 StrategicDirective。
统治者 agent 可以综合部长建议，形成最终战略目标。
保留未来“玩家扮演将领”的接口，但 v0.5 不实现完整将领生涯模式。
交付物：
统治者 agent 配置。
玩家国家级命令入口的数据模型。
统治者决策 JSON schema。
测试：玩家命令能进入仲裁层并下发给战区/将领。
依赖：
Agent C 的 DirectiveBoard。
Agent D 的部长建议。
v0.4 的聊天命令能力。
Agent F：内阁/Agent UI 与日志
负责让玩家看懂多 agent 系统在干什么。
工作内容：
新增内阁面板：统治者、部长、元帅、将领列表。
显示每个 agent 的角色、忠诚/信任、当前建议、最近决策。
显示部长会议日志。
显示已批准战略指令和被拒绝建议。
显示某条战略指令如何下发到前线/将领。
提供调试视图：原始 JSON、解析结果、仲裁结果。
交付物：
CabinetView / AgentCouncilView。
DirectiveBoardView。
AgentDecisionLogView 扩展。
UI smoke test 或手动验收步骤。
依赖：
Agent A/C/D/E 的模型和日志数据。
不改变核心规则。
推荐开发顺序
Agent A 先做角色模型和配置。
Agent B 做分层摘要。
Agent C 做战略指令和仲裁。
Agent D 做部长 Mock 决策。
Agent E 接入统治者和玩家国家级命令。
Agent F 最后做 UI 和日志展示。
可以并行的部分：
A 和 B 可以先约定接口后并行。
D 可以在 C 的 StrategicDirective 初稿稳定后并行。
F 应最后做，避免 UI 反复追模型变更。
v0.5 明确不做
不做完整经济模拟。
不做完整科技树。
不做真实外交系统。
不做真实政变系统。
不做完整内购。
不让部长直接改资源、单位、地图或科技。
不让统治者 agent 绕过 Command / Directive 管线。
不让多个 agent 同时直接控制同一个单位。
Acceptance Tests
能从 JSON 加载一个国家内阁，包含统治者、4 个部长、1 个元帅、至少 1 个将领。
每个 agent 都能生成与自己角色匹配的上下文摘要。
部长 MockAI 能输出结构化建议，并被 DirectiveBoard 接收。
仲裁器能处理至少 2 条冲突建议，并记录批准/拒绝原因。
玩家国家级命令能转换成 StrategicDirective。
被批准的战略指令能下发给战区/将领层，但不直接移动单位。
UI 能展示内阁成员、建议、批准指令、拒绝原因和原始 JSON。
关闭真实 LLM 时，v0.5 仍可完整跑通。
Assumptions
v0.5 建立“国家与部长 agent 骨架”，不实现完整大战略数值系统。
v0.5 默认使用 MockAI，真实本地 LLM 只作为可替换 provider。
阶段 0-4 的 GameState、Command、DecisionProvider、日志系统已经可用。
如果 v0.3 前线系统尚未完成，v0.5 的战略指令先下发为日志和计划，不强制驱动真实前线自动执行。
