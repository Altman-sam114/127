你现在要为一个新项目复刻一套“人类目标 -> Agent A 设计提示词 -> Agent B 实现测试 -> Agent C 验收并更新核心逻辑文档 -> 人工复核 -> 下一轮”的迭代工作流。

目标不是立刻开发功能，而是先为项目建立一套可长期迭代的 Agent 协作系统、记忆文档、测试规范、核心流程文档和提示词目录。

## 任务目标

请在当前项目中创建或更新以下结构：

- `AGENTS.md`：项目入口记忆、总览、基本规则、Agent 工作流。
- `update_log.md`：版本更新记录、历史决策、完成事项、遗留问题。
- `md/prompt/`：每轮 Agent A 输出详细实现提示词的位置，按版本号管理。
- `md/test/test.md`：测试规范、测试分层、命令、触发条件、当前基线。
- `md/flow/flow.md`：项目核心逻辑文档，记录当前真实架构和运行流程。
- `md/flow/flowchart.md`：flow.md的可视化mermaid 逻辑图，描述项目核心数据流、执行流，先写好flow.md，再编写本流程图，用于给人工看实时逻辑（每个流程逻辑块注释中文解释，每张图前加入通俗读图说明，方便快速看懂逻辑）。
- 若用户明确要求 `upload_log.md` 或 `mc/test`，先确认是否为命名偏好；若是，则按用户命名创建；若不是，则使用标准命名 `update_log.md` 和 `md/test/test.md`。

不要写空泛模板。请先阅读项目现状，再写适合本项目的文档。

## 总体工作流

这个项目以后按三类 Agent 循环迭代：

### 人工

人工提出实现目标。对于精做项目，人工可以同时给出：

- 目标功能。
- 算法框架。
- 禁止项。
- 验收标准。
- 性能要求。
- UI/交互要求。
- 测试要求。

人工提出想做到的目标，通过`AGENTS.md`将`update_log.md`、`md/flow/flow.md`、`md/test/test.md` 和相关上下文记忆交给 Agent A。

### Agent A：目标分析与提示词

Agent A 负责思考“这个目标应该怎么做”，默认不直接写代码。

Agent A 必须：

1. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`。
2. 阅读与任务相关的源码、测试、已有文档。
3. 明确本轮目标、非目标、边界、依赖、风险和验收标准。
4. 设计实现方案，包括：
   - 需要改哪些模块。
   - 数据流/状态流怎么变化。
   - 是否需要新增接口、模型、测试。
   - 哪些旧逻辑必须保持不变。
   - 哪些行为必须通过测试锁住。
5. 为本轮提示词确定版本号：人工指定版本时按人工指定；人工未指定时由 Agent A 自动分配，从 `v0.1` 开始递增。小任务、修复、优化按 `v0.2`、`v0.3` 这类小版本推进；大任务、架构阶段或重要里程碑可新开 `v1.0`、`v2.0` 这类大版本。
6. 写一份给 Agent B 的详细实现提示词，放到 `md/prompt/v0（简要标题）/v0.1（简要说明）.md` 这种格式的版本目录中，例如 `md/prompt/v0（基础框架）/v0.1（建立项目骨架）.md`。

Agent A 输出的提示词必须包含：

- 版本号。
- 版本分配依据：人工指定或 Agent A 自动判断。
- 背景。
- 目标。
- 非目标。
- 当前架构依据。
- 实现步骤。
- 关键文件。
- 测试要求。
- 文档更新要求。
- 验收标准。
- 风险和禁止项。

### Agent B：实现与测试

Agent B 负责按 Agent A 写的提示词实现。

Agent B 必须：

1. 阅读 Agent A 提示词。
2. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/test/test.md`。
3. 阅读相关源码和测试。
4. 小步实现，不做无关重构。
5. 根据任务新增或修改测试。
6. 按 `md/test/test.md` 选择测试层级。
7. 运行测试并记录具体命令和结果。
8. 更新必要文档。
9. 输出实现结果，包括：
   - 改了什么。
   - 关键文件。
   - 测试命令和结果。
   - 未跑测试及原因。
   - 已知风险。
   - 后续建议。

Agent B 不得：

- 绕过项目核心规则直接改状态。
- 擅自扩大范围。
- 删除旧实现。
- 用“已验证”代替具体测试结果。
- 伪造测试通过。
- 回滚用户或其他 Agent 的改动。

### Agent C：验收与核心逻辑更新

Agent C 负责验收 Agent B 的结果，并更新项目核心逻辑文档。

Agent C 必须：

1. 阅读 Agent B 输出。
2. 查看实际 diff。
3. 核对测试结果。
4. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/test/test.md`。
5. 判断实现是否满足 Agent A 提示词和人工目标。
6. 检查：
   - 架构边界是否被破坏。
   - 测试是否足够。
   - 文档是否同步。
   - 是否有未说明风险。
7. 基于 Agent B 新实现的功能/结构，更新 `md/flow/flow.md`。
8. 更新 `md/flow/flowchart.md` 的 mermaid 图。
9. 如形成正式版本或重要历史事项，更新 `update_log.md`。
10. 输出验收结论：
   - 通过/不通过。
   - 问题清单。
   - 已更新文档。
   - 建议下一步。

最后回到人工，由人工复核、验收、继续下一轮。
