# 二战 AI 战略战棋游戏规划总结

## 1. 项目定位

目标是制作一款 iOS 回合制二战策略游戏，结合战棋、大战略和角色扮演系统。

参考方向：

- 《钢铁雄心4》：大战略、省份占领、前线、补给、生产、国家管理。
- EasyTech《钢铁命令》：战役式推进、将领、战术操作。
- 《世界征服者4》：移动占领、将领养成、轻量化移动端策略体验。

核心创新：

- 本地部署 LLM 驱动游戏 AI。
- 每个将领、元帅、国家统治者、部长后续都可以是一个 agent。
- agent 根据视野、战况、性格、职位和历史背景生成结构化命令。
- 游戏规则系统负责校验命令是否合法，再转换成游戏内行动。
- 玩家可以通过聊天框直接命令将领，也可以扮演统治者、元帅或将领。

核心原则：

- v0 先做小型测试板，不直接做完整大作。
- LLM 负责高层意图和计划，不直接控制每个单位的底层细节。
- 规则、地图、命令、AI、UI 必须解耦，方便长期扩展。

## 2. 地图方案分析

### 2.1 可选方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 方格 | 最容易实现 | 战线、包围和地形边界不自然 | 不推荐 |
| 纯六角格 | 战棋感强，移动、包围、侧翼规则清晰 | 大地图格子数量过多，前线和补给管理复杂 | 适合 v0，不适合作为长期唯一底层 |
| 纯省份图 | 最接近《钢铁雄心4》，适合占领、补给、前线和战略 AI | 局部战术感较弱，早期实现和表现更复杂 | 长期适合 |
| 省份图底层 + 六角格表现 | 兼顾早期可实现性和长期扩展 | 架构上必须一开始分清规则区块与显示格子 | 最推荐 |

### 2.2 省份图是什么

省份图不是规则网格，而是由一个个可占领的区块节点组成。

示例：

```text
阿登森林区块
- 当前控制方：德国
- 原始归属：比利时
- 地形：森林
- 基建：2
- 补给：低
- 相邻区块：列日、巴斯托涅、卢森堡、色当
- 是否城市：否
- 是否要塞：否
```

游戏规则关心的是：

- 区块属于谁；
- 谁控制该区块；
- 哪些区块相邻；
- 部队能否从一个区块移动到另一个区块；
- 补给线是否连通；
- 前线边界在哪里；
- 城市、工厂、机场、港口、资源是否挂在该区块上。

### 2.3 推荐结论

长期建议使用：

```text
省份图 / 区块图作为规则底层
六角格作为 v0 测试板表现方式或早期简化区块
```

v0 阶段可以先让：

```text
一个六角格 = 一个区块
```

后续升级为：

```text
一个省份 = 一个不规则多边形区块
或
一个省份 = 多个显示格组成的战略区块
```

重要命名建议：

- 规则层不要直接命名为 `HexCell`。
- 优先使用 `ProvinceNode`、`MapRegion`、`RegionNode`。
- 六角格只作为一种地图表现方式或坐标实现。

## 3. v0 初步测试板范围

v0 的目标不是完整游戏，而是验证核心闭环。

需要验证：

1. 六角格/区块地图是否可玩。
2. 部队移动、攻击、占领是否成立。
3. 补给、包围、侧翼攻击是否能形成有意义的战术。
4. 本地 LLM 将领 agent 能否读取战场摘要并输出结构化命令。
5. 命令能否被规则系统校验并执行。
6. 玩家能否查看 AI 决策过程和战斗结果。

### 3.1 v0 建议战役

小型“阿登测试战场”。

阵营：

- 德国
- 盟军

地图：

- 小尺寸六角格地图。
- 包含森林、平原、城市、道路、河流、要塞。
- 设置少量关键城市和交通节点。

单位：

- 坦克
- 摩托化步兵
- 步兵
- 炮兵

玩法：

- 玩家或 AI 控制若干师。
- 师可以移动、攻击、防守、撤退。
- 部队进入敌方区块后占领该区块。
- 占领关键城市或切断补给可以获得胜利。

### 3.2 v0 暂不做

- 完整经济系统。
- 完整科技树。
- 空军实体化。
- 海军。
- 天气。
- 多国家外交。
- 内购。
- 大规模将领养成。
- 完整省份图美术。
- 多层级 agent 政治系统。

这些系统应预留接口，但不在 v0 实现。

## 4. 核心战斗与单位系统

### 4.1 基础单位

v0 使用“师”作为基本作战单位。

师可以包含混编权重：

```text
装甲师
- 坦克：60%
- 摩托化步兵：30%
- 炮兵：10%
```

```text
步兵师
- 步兵：80%
- 炮兵：20%
```

### 4.2 基础属性

每个师至少包含：

- `strength`：兵力和装备完整度。
- `organization`：组织度，表示战斗秩序、疲劳和士气。
- `supply`：补给状态。
- `movement`：本回合剩余移动力。
- `vision`：视野。
- `attack`：攻击能力。
- `defense`：防御能力。

### 4.3 组织度与撤退

组织度应尽早加入，至少在 v0.1 加入。

规则建议：

- `strength` 低代表人员和装备损失，恢复慢，需要补员和装备。
- `organization` 低代表部队混乱，恢复相对快。
- 战斗失败优先导致组织度崩溃，而不是直接全灭。
- 组织度归零时部队强制撤退。
- 没有撤退路线时，部队可能投降或被歼灭。
- 被包围时组织度恢复降低，补员停止或大幅减弱。

### 4.4 包围、补给和侧翼

建议 v0 就加入简化版：

- 如果部队无法连接到己方补给源，则进入低补给状态。
- 低补给降低攻击、防御、移动和组织度恢复。
- 多方向攻击同一目标时，增加组织度伤害。
- 从侧后方攻击时，增加伤害或降低敌方防御。
- 被包围单位不能正常撤退。

## 5. 前线系统规划

前线是长期核心系统，建议从 v0.3 开始实现。

基本定义：

```text
前线 = 己方控制区块与敌方控制区块相邻的边界集合
```

前线系统能力：

- 自动识别敌我边界。
- 玩家或 agent 可以把集团军分配到某条前线。
- 系统自动让部队沿前线布防。
- 优先填补缺口。
- 优先防守城市、河流、要塞、山地。
- 补给差的区块减少驻军密度。
- 装甲部队不平均摊开，应集中在突破点。
- 前线过长而部队不足时，提示战线过度拉长。

进攻计划：

- 玩家或元帅 agent 选择突破方向。
- 将领 agent 负责局部执行。
- 系统把战略计划拆成具体移动和攻击命令。

## 6. LLM Agent 系统

### 6.1 v0 agent 范围

v0 只实现一个或少量将领 agent。

例如：

```text
古德里安 agent
- 国家：德国
- 职位：将领
- 兵种偏好：装甲、摩托化
- 性格：进攻性强，偏好突破、包围、快速推进
- 风险承受：高
- 指挥范围：若干个师
```

### 6.2 输入给 LLM 的信息

LLM 不应读取完整地图真相，只能读取摘要。

输入应包含：

- 将领身份和性格。
- 当前任务。
- 己方可见区块。
- 己方单位状态。
- 可见敌军。
- 补给状况。
- 地形和关键目标。
- 上回合战斗结果。
- 可用命令类型。

### 6.3 LLM 输出

LLM 必须输出结构化 JSON。

示例：

```json
{
  "intent": "breakthrough",
  "summary": "集中装甲力量突破森林南侧，切断敌军补给。",
  "commands": [
    {
      "type": "move",
      "unitId": "ger_panzer_1",
      "targetRegionId": "bastogne_west"
    },
    {
      "type": "attack",
      "unitId": "ger_panzer_2",
      "targetRegionId": "bastogne"
    }
  ]
}
```

### 6.4 命令执行原则

LLM 只能提出建议，不能绕过规则。

流程：

```text
战场状态摘要
-> 将领 prompt
-> 本地 LLM 输出 JSON
-> JSON 解析
-> 命令合法性校验
-> 非法命令修复或丢弃
-> 规则系统执行
-> 生成战报和 agent 决策日志
```

如果 LLM 输出非法命令：

- 第一次尝试自动修复。
- 修复失败则丢弃非法命令。
- 必要时使用 fallback 规则 AI。
- 决策日志记录错误原因。

## 7. 长期角色扮演与多 Agent 方向

后续可以升级为多层级 agent 系统。

角色层级：

- 国家统治者 agent：总体战略、政治路线、战争目标。
- 元帅 agent：战区指挥、前线计划、集团军群调度。
- 将领 agent：局部战役执行、师级指挥。
- 部长 agent：经济、军工、科技、外交、情报。
- 玩家：可扮演统治者、元帅、将领或前线指挥官。

玩家交互方式：

- 通过聊天框命令古德里安等将领。
- 作为统治者下达战略目标。
- 作为元帅安排前线和攻势。
- 作为将领执行上级命令。
- 也可以直接微操部队。

角色关系系统：

- 忠诚度。
- 信任度。
- 威望。
- 派系。
- 抗命概率。
- 任免和晋升。
- 政变风险。

示例：

```text
玩家作为统治者：
“古德里安，三周内突破阿登，切断盟军北翼。”

古德里安 agent：
“我建议集中第1、第2装甲集团，绕过坚固城市，优先夺取交通节点。”
```

如果玩家频繁越级微操，可能导致：

- 将领不满。
- 执行效率下降。
- 延迟执行。
- 抗命。
- 派系矛盾。

## 8. 性能与复杂度控制

本地 LLM 是核心特色，但必须限制调用规模。

优化原则：

1. 不让每个单位调用 LLM。
2. 每个将领、元帅或战区按回合或关键事件调用。
3. LLM 读取摘要，不读取完整地图。
4. 小决策由规则 AI 处理。
5. LLM 负责战略意图、作战计划和角色表达。
6. 前线自动排兵使用算法，不让 LLM 逐个安排所有单位。
7. agent 记忆分为短期战场记忆和长期角色记忆。
8. 对每个 agent 设置 token 和调用频率预算。
9. 命令执行必须异步，并允许 fallback AI 接管。

推荐决策链：

```text
玩家/Agent 意图
-> 作战计划 JSON
-> 规则系统校验
-> 自动规划器拆解
-> 单位命令
-> 执行
-> 战报摘要
```

不推荐：

```text
LLM 直接控制每个单位每一步
```

## 9. 工程架构建议

iOS 技术栈建议：

- Swift
- SwiftUI
- SpriteKit

核心模块：

- `GameState`：完整游戏状态。
- `MapState`：地图区块、控制权、地形、补给。
- `RegionNode` / `ProvinceNode`：规则层地图区块。
- `Division`：师级单位。
- `GeneralAgent`：将领 agent。
- `Command`：结构化命令。
- `RuleEngine`：规则校验和执行。
- `TurnManager`：回合流程。
- `LLMClient`：本地 LLM 接口。
- `MockAI`：无 LLM 时的 fallback AI。
- `BattleLog`：战斗和 AI 决策日志。
- `FrontendPlanner`：前线和进攻计划系统，后续加入。

架构要求：

- 游戏规则逻辑与 UI 分离。
- 地图显示方式与规则区块分离。
- LLM 接口可替换。
- 没有 LLM 时可以用 MockAI 跑通。
- 所有命令先校验再执行。
- 每回合可以查看日志。

## 10. 长期路线图

### v0：六角格测试板

- 小型阿登测试地图。
- 六角格显示，底层按 `RegionNode` 建模。
- 德军 vs 盟军。
- 少量陆军单位。
- 移动、攻击、占领、基础补给。
- MockAI + 一个将领 agent。
- 结构化命令解析、校验、执行。
- AI 决策日志 UI。

### v0.1：组织度、撤退与补员

- 加入 `strength` 和 `organization`。
- 组织度崩溃后撤退。
- 补员和组织度恢复。
- 包围导致补给和恢复惩罚。
- 更清晰的战斗日志。

### v0.2：省份图底层

- 把地图规则层正式抽象为 `ProvinceNode` / `RegionNode`。
- 六角格变成显示方式或早期地图数据格式。
- 区块控制权、核心区、占领区。
- 城市、工厂、资源挂到区块上。
- 为经济和政治系统预留数据结构。

### v0.3：前线系统

- 自动识别前线。
- 集团军分配前线。
- 自动填补缺口。
- 攻势方向和突破点。
- 元帅 agent 制定战区计划。
- 将领 agent 执行局部计划。

### v0.4：聊天命令与服从系统

- 玩家通过聊天框命令将领。
- 将领根据性格、忠诚和战况回应。
- 命令可能被质疑、拖延或抗命。
- 玩家可以微操前线，但会影响将领满意度和信任度。

### v0.5：国家与部长 agent

- 国家统治者 agent。
- 军工部长、陆军部长、外交部长、情报部长。
- 经济、科技、生产和外交建议。
- 高层 agent 下发战略目标。

### v1.0：大战略原型

- 战役模式。
- 征服模式。
- 经济：资源、工厂、税收。
- 人力系统。
- 科技和装备生产。
- 空军实体化。
- 简化海军。
- 天气。
- 多国家、多战区、多 agent 协作。
- 将领购买或养成系统。
- 部队、将领、地图美术资源。

当前阶段只做 v0 初步测试板：
