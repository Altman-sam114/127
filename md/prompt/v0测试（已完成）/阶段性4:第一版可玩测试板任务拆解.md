# 阶段性 4：第一版可玩测试板任务拆解

本文档用于指导后续 agent 逐步实现 iOS v0 测试板。目标不是一次性完成完整大战略游戏，而是构建一个可运行、可观察、可扩展的六角格战棋原型。

默认技术栈：

- 平台：iOS
- 语言：Swift
- UI：SwiftUI
- 地图渲染：SpriteKit
- AI：先接 MockAI，再预留本地 LLM 连接
- 场景：阿登测试战场
- 阵营：德国 vs 盟军

核心工程原则：

- 玩家和 AI 都只提交命令，不直接修改 `GameState`。
- 所有状态变化必须经过 `RulesEngine` 校验和执行。
- MockAI 与未来本地 LLM 共用同一个 `DecisionProvider` 接口。
- v0 只做陆军战棋闭环，空军、海军、天气、科技、经济等系统只保留接口位置。

## 2. 六角格地图渲染

### 目标

在 SpriteKit 中渲染一个固定尺寸的六角格地图，为后续地形、单位、移动范围和战争迷雾提供显示基础。

### 涉及模块

- `BoardScene`
- `HexRenderer`
- `HexLayout`
- `GameState`
- `HexTile`

### 输入输出

输入：

- 地图宽高，例如 `11 x 9`。
- `HexTile` 列表。
- 每个 hex 的 axial 坐标。

输出：

- 屏幕上的六角格网格。
- 每个格子可以根据坐标定位。
- 支持从屏幕点击位置反查 hex 坐标。

### 完成标准

- 地图中每个 hex 都被正确绘制。
- hex 之间没有明显错位。
- 支持不同屏幕尺寸下居中缩放。
- 点击任意 hex 可以在日志中输出对应 `q, r`。
- 渲染层和规则层解耦，SpriteKit 不直接计算战斗规则。

### 简单测试方法

- 在 Simulator 中观察地图是否完整显示。
- 点击左上、中间、右下格，确认坐标输出正确。
- 写单元测试验证 `HexLayout.hexToPixel` 和 `HexLayout.pixelToHex` 基本互逆。

## 4. 单位数据与显示

### 目标

实现陆军单位数据结构，并在地图上显示双方单位、血量、阵营和补给状态。

### 涉及模块

- `CombatUnit`
- `DivisionTemplate`
- `DivisionComponent`
- `UnitType`
- `Faction`
- `SupplyState`
- `UnitRenderer`
- `BoardScene`

### 输入输出

输入：

- 单位列表。
- 单位坐标。
- 单位阵营。
- 单位 hp 和补给状态。

输出：

- 地图上显示单位 marker。
- 单位 marker 显示阵营颜色。
- 单位 marker 显示简短兵种标记。
- 单位血量和补给状态可见。

单位类型：

| 单位类型 | 说明 |
|---|---|
| `tank` | 坦克/装甲主力 |
| `motorizedInfantry` | 摩托化步兵 |

混编师结构：

```swift
struct DivisionComponent {
    let type: UnitType
    let weight: Double
}
```

v0 单位在地图上仍是一个师，师属性由兵种权重汇总。

### 完成标准

- 德军和盟军单位能同时显示。
- 同一个 hex 最多显示一个单位。
- 单位 hp 改变后 UI 会更新。
- 单位补给状态改变后 UI 有可见标记。
- 单位数据不依赖 SpriteKit 节点本身。

### 简单测试方法

- 修改某个单位补给状态为 `lowSupply` 或 `encircled`，确认 UI 标记变化。
- 单元测试验证混编师权重总和必须接近 `1.0`。

## 5. 选择单位与移动范围

### 目标

实现玩家选择己方单位、显示合法移动范围、点击目标 hex 后提交移动命令并执行。

### 涉及模块

- `SelectionController`
- `MovementSystem`
- `RulesEngine`
- `Command`
- `BoardScene`
- `GameState`

### 输入输出

输入：

- 当前玩家阵营。
- 点击的 hex 坐标。
- 当前单位位置。
- 单位移动力。
- 地形移动消耗。
- 敌方单位和控制区。

输出：

- 当前选中单位。
- 合法移动范围集合。
- 移动命令。
- 更新后的单位坐标。
- 行动日志。

### 完成标准

- 玩家只能选择当前阵营的单位。
- 已行动单位不能再次移动。
- 不能移动到敌方单位所在格。
- 不能移动到己方单位所在格。
- 进入敌方控制区后移动结束。
- 移动后单位 `hasActed = true`。
- 移动路径和目标必须经过规则校验。

### 简单测试方法

- 点击盟军单位，显示移动范围。
- 点击范围内平原格，单位移动成功。
- 点击范围外格，单位不移动并记录失败原因。
- 点击敌方单位所在格，移动失败。
- 单元测试验证山地消耗导致可达范围变小。

## 6. 攻击与伤害

### 目标

实现攻击、反击、射程、地形防御、侧翼/绕后加成和单位死亡移除。

### 涉及模块

- `CombatSystem`
- `RulesEngine`
- `DamageCalculator`
- `LineOfSightPolicy`
- `CombatLog`
- `GameState`

### 输入输出

输入：

- 攻击单位 id。
- 防守单位 id。
- 攻击者属性。
- 防守者属性。
- 地形防御修正。
- 射程。
- facing 方向。
- 补给状态。

输出：

- 攻击是否合法。
- 攻击伤害。
- 反击伤害。
- 更新后的 hp。
- 单位死亡结果。
- 战斗日志。

攻击规则：

- 非炮兵射程为 `1`。
- 炮兵射程为 `2`。
- 攻击目标必须可见。
- 攻击者必须未行动。
- 防守者存活且射程覆盖攻击者时可以反击。
- 炮兵被近战攻击时不能反击。

### 完成标准

- 相邻攻击可执行。
- 炮兵可 2 格攻击。
- 地形防御修正参与伤害计算。
- 侧翼攻击有伤害加成。
- 绕后攻击有更高伤害加成。
- 目标 hp 到 `0` 后从地图移除。
- 攻击后攻击者 `hasActed = true`。

### 简单测试方法

- 步兵攻击相邻步兵，确认 hp 降低。
- 炮兵攻击 2 格外目标，确认成功。
- 步兵攻击 2 格外目标，确认失败。
- 攻击要塞内单位，确认伤害低于平原。
- 从背后攻击单位，确认伤害高于正面。

## 7. 回合切换

### 目标

实现德军 AI 回合、盟军玩家回合、结算阶段和胜负检查的基础流程。

### 涉及模块

- `TurnSystem`
- `GamePhase`
- `RulesEngine`
- `VictorySystem`
- `GameState`
- `UIControls`

### 输入输出

输入：

- 当前回合数。
- 当前阶段。
- 所有单位行动状态。
- 当前胜利状态。

输出：

- 新阶段。
- 新回合数。
- 重置后的单位行动状态。
- 回合日志。

回合流程：

```text
GermanAI -> AlliedPlayer -> Resolution -> GermanAI
```

### 完成标准

- 新回合开始时对应阵营单位 `hasActed = false`。
- 达成胜利条件后进入 `finished`。

### 简单测试方法

- 连续点击结束回合，确认回合数递增。
- 移动一个盟军单位后结束回合，下一次盟军回合该单位可再次行动。
- 设置胜利条件为已满足，确认 phase 进入 `finished`。

## 8. 补给与包围

### 目标

实现补给线判定、低补给惩罚和包围状态。

### 涉及模块

- `SupplySystem`
- `ZoneOfControlSystem`
- `Pathfinding`
- `RulesEngine`
- `GameState`

### 输入输出

输入：

- 单位坐标。
- 阵营补给源。
- 地图控制权。
- 敌方单位位置。
- 敌方控制区。
- 地形和道路。

输出：

- 单位补给状态：`supplied`, `lowSupply`, `encircled`。
- 有效属性修正。
- 包围损耗日志。

补给规则：

- 单位到己方补给源存在有效路径，则 `supplied`。
- 无有效路径但仍有至少 2 个安全撤退邻格，则 `lowSupply`。
- 无有效路径且安全撤退邻格少于 2 个，则 `encircled`。
- 敌方单位所在格不可通过。
- 敌方控制城市/要塞不可通过。
- 敌方控制区不可通过，除非该格有己方单位。

### 完成标准

- 每回合结算时重算所有单位补给状态。
- 低补给单位移动、攻击、防御受到惩罚。
- 被包围单位额外受到损耗。
- 补给状态在 UI 上可见。
- 补给判定不依赖渲染层。

### 简单测试方法

- 创建一条道路补给线，确认单位 `supplied`。
- 用敌军切断道路，确认单位变为 `lowSupply`。
- 用敌军控制区压住周围出口，确认单位变为 `encircled`。
- 单元测试验证 `encircled` 单位攻击力低于正常状态。

### 目标

实现每个阵营独立的可见区域、已探索区域和未知区域，并限制玩家与 AI 对敌军信息的读取。

输入：

- 当前阵营。
- 己方单位位置。
- 单位视野。
- 地形视野修正。

输出：

- `visible` hex 集合。

视野建议：

| 单位类型 | 视野 |
|---|---:|

## 10. MockAI

### 目标

实现德军古德里安 MockAI，让 AI 在不接真实 LLM 的情况下输出结构化决策。

### 涉及模块

- `DecisionProvider`
- `MockDecisionProvider`
- `AgentContext`
- `AgentDecisionEnvelope`
- `AgentOrder`
- `RulesEngine`
- `GameState`

### 输入输出

输入：

- `AgentContext`
- 德军可行动单位。
- 可见盟军单位。
- 目标城市。
- 最近日志。
- 玩家对德军的可选文本指令。

输出：

- `AgentDecisionEnvelope`
- 原始 JSON 字符串。
- 决策意图 `intent`。
- 命令列表。

MockAI 行为：

- 装甲优先沿道路接近 `Bastogne`。
- 炮兵优先攻击要塞、城市、低 hp 敌军。
- 如果有低 hp 可见敌军，优先集火。
- 低补给单位优先 `resupply` 或后撤。
- 被包围单位优先脱离包围。

### 完成标准

- MockAI 每个德军回合稳定产出 JSON。
- JSON 能 decode 为 `AgentDecisionEnvelope`。
- 命令不直接修改游戏状态。
- MockAI 可以在没有可攻击目标时生成移动或 hold。
- MockAI 输出包含每条命令的 `reason`。

### 简单测试方法

- 构造敌军在炮兵射程内，确认 MockAI 生成 attack。
- 构造装甲到目标城市有道路，确认 MockAI 生成 move。
- 构造低补给单位，确认 MockAI 生成 resupply 或 hold。
- 单元测试验证 MockAI 输出能被 JSON decode。

## 11. 本地 LLM 连接

### 目标

为后续接入本地 LLM 预留实现，并提供可关闭的第一版 HTTP 连接器。v0 默认仍使用 MockAI。

### 涉及模块

- `DecisionProvider`
- `LocalLLMDecisionProvider`
- `LLMClient`
- `LLMConfiguration`
- `AgentContextSerializer`
- `AgentDecisionParser`

### 输入输出

输入：

- `AgentContext`
- 本地 LLM endpoint，例如 Ollama 或 LM Studio。
- prompt 文本。
- JSON schema 或格式说明。

输出：

- LLM 原始响应文本。
- 尝试解析后的 `AgentDecisionEnvelope`。

建议配置：

```swift
struct LLMConfiguration {
    var isEnabled: Bool
    var endpoint: URL
    var modelName: String

}
```

### 完成标准

- v0 默认不启用本地 LLM。
- 打开配置后，可以向本地 HTTP endpoint 发送请求。
- 网络失败、超时、JSON 错误不会导致游戏崩溃。
- LLM 失败时可以 fallback 到 MockAI。
- LLM provider 与 MockAI provider 使用同一协议。

### 简单测试方法

- 在未启用 LLM 时确认使用 MockAI。
- 配置一个不存在 endpoint，确认日志显示失败并 fallback。
- 用本地假服务返回固定 JSON，确认可以解析。
- 单元测试验证非法 LLM 响应不会修改 `GameState`。

## 12. 将领 agent prompt

### 目标

定义古德里安 agent 的第一版 prompt 模板，约束其只输出合法 JSON 命令，不输出散文。

### 涉及模块

- `AgentPromptBuilder`
- `AgentProfile`
- `AgentContext`
- `AgentDecisionEnvelope`
- `LocalLLMDecisionProvider`

### 输入输出

输入：

- agent 人设。
- 当前局势摘要。
- 可用命令类型。
- JSON 输出格式。
- 游戏规则摘要。
- 玩家文本指令。

输出：

- prompt 字符串。
- 可发送给 LLM 的消息体。

Prompt 必须包含：

- 角色：德军将领古德里安。
- 战术偏好：装甲突破、快速穿插、道路优先、集中火力。
- 约束：不能违反规则、不能使用不可见敌军信息、不能生成不存在单位。
- 输出：只能输出 JSON。
- 命令类型：`move`, `attack`, `hold`, `resupply`。

### 完成标准

- Prompt 可以从 `AgentContext` 稳定生成。
- Prompt 明确要求 JSON schema。
- Prompt 明确禁止解释性自然语言输出。
- Prompt 中包含当前可见单位和目标。
- Prompt 与 MockAI 解耦，主要给 `LocalLLMDecisionProvider` 使用。

### 简单测试方法

- 构造一个 `AgentContext`，打印 prompt，人工检查字段完整。
- 确认 prompt 不包含不可见敌军。
- 确认 prompt 中包含命令 schema。
- 用固定 LLM 响应测试 parser。

## 13. JSON 命令解析与校验

### 目标

实现从 AI JSON 到可执行规则命令的解析、校验和执行管线。

### 涉及模块

- `AgentDecisionParser`
- `CommandValidator`
- `CommandExecutor`
- `RulesEngine`
- `AgentDecisionEnvelope`
- `AgentOrder`
- `CommandValidationResult`
- `GameState`

### 输入输出

输入：

- AI 原始 JSON。
- 当前 `GameState`。
- 当前 AI 阵营。
- 当前回合。

输出：

- decode 后的 `AgentDecisionEnvelope`。
- 每条命令的校验结果。
- 已执行命令列表。
- 被拒绝命令列表。
- 更新后的 `GameState`。

校验规则：

- `schemaVersion` 必须支持。
- `agentId` 必须匹配当前 agent。
- `turn` 必须匹配当前回合。
- `unitId` 必须存在。
- 单位必须属于当前 AI 阵营。
- 单位必须未行动。
- `move` 必须有合法 `to`。
- `attack` 必须有合法 `targetUnitId`。
- 目标必须在射程内且可见。
- `hold` 和 `resupply` 不需要目标。

### 完成标准

- 非法 JSON 不会崩溃。
- 部分命令非法时，合法命令仍可执行。
- 每条命令都有明确通过或失败原因。
- 执行顺序与 JSON 中顺序一致。
- AI 无法绕过规则系统修改状态。

### 简单测试方法

- 输入合法 move JSON，确认单位移动。
- 输入敌方单位 id，确认命令被拒绝。
- 输入越界坐标，确认命令被拒绝。
- 输入重复命令，确认第二条因 `hasActed` 被拒绝。
- 输入 malformed JSON，确认状态不变且日志记录错误。

## 15. 阿登测试场景

### 目标

实现一张小型固定战役地图，验证移动、攻击、占领、补给、包围、战争迷雾和 AI 决策。

### 涉及模块

- `ArdennesScenario`
- `ScenarioLoader`
- `GameStateFactory`
- `VictorySystem`
- `BoardScene`
- `RulesEngine`

### 输入输出

输入：

- 固定地图定义。
- 城市和要塞位置。
- 道路和河流定义。
- 双方单位初始部署。
- 胜利条件。

输出：

- 可玩的阿登测试战场。
- 初始 `GameState`。
- 场景名称、目标、回合数。

建议地图：

- 尺寸：`11 x 9`
- 德军从东侧进入。
- 盟军守中西部。
- 关键城市：
  - `Bastogne`
  - `St. Vith`
  - `Houffalize`
- 要塞：
  - `Bastogne Fortress`
- 补给源：
  - 德军东侧补给源
  - 盟军西侧补给源

建议初始单位：

德国：

- 第 1 装甲师
- 第 2 摩步师
- 第 3 步兵师
- 第 4 炮兵师

盟军：

- 第 101 守备步兵师
- 第 2 步兵师
- 第 3 反装甲步兵师
- 第 4 炮兵师

胜利条件：

- 德军在第 8 回合结束前占领 `Bastogne` 并保持 1 个完整回合。
- 或德军消灭 3 个盟军单位。
- 盟军在第 8 回合结束时仍控制 `Bastogne`。
- 或盟军消灭 3 个德军单位。
- 或盟军切断所有德军装甲单位补给并维持 1 个完整回合。

### 完成标准

- 场景可以一键加载。
- 地图包含平原、森林、山地、城市、要塞、河流、道路。
- 双方单位初始位置合理，不会第一回合直接结束游戏。
- 至少能完成 8 回合。
- 德军 AI 能尝试进攻 `Bastogne`。
- 盟军玩家有防守、反击、切补给线的空间。

### 简单测试方法

- 启动 App 后自动加载阿登测试场景。
- 手动完成 3 回合，确认没有卡死。
- 点击德军 AI 执行，确认至少有移动或攻击。
- 切断一支德军单位补给，确认状态变化。
- 让德军占领 `Bastogne`，确认胜利条件开始计数。
