# v0.42 将领指派器与地图编辑器扩展

## 阶段目标
在 0.41 我们有了实体将领数据，并能在硬编码下给战区分配特定将领。本阶段（0.42）的目标是建立**正规的将领生命周期调度体系（Dispatcher）**：允许在地图编辑器中直接指定开局将领；并在前线/战区动态变化（分裂、消亡）时，自动根据规则进行新老将领的替补与调度，彻底告别硬编码。

## 详细任务拆解

### 1. 扩展地图编辑器 (Tiled/JSON 地图解析)
- **更新地图格式**：修改当前的地图场景文件流或 `MapRegion` 定义，允许在初始的特定的 Region（属于某个初始战区）中附带一个额外的元数据：`assignedGeneralId`。
- **解析接入**：当地图被加载并初始化 `TheaterState` / `WarDeploymentState` 时，将这些 `generalId` 绑定到最初生成的战区（`FrontZoneId`）上。
- （*这使你可以直接用现在的 Tiled 编辑器给地图埋点，开局直接产生一位带古德里安的特定战区*）

### 2. 编写通用将领指派器 (GeneralDispatcher)
- **作用**：这是一个位于回合级别（或者战区生成周期级别）的核心调度中心。
- **状态维护**：需要知道当前本阵营有哪些**可用将领（在后备池）**，目前有哪些挂在战区上的**现役将领**。
- **事件响应**：
  - **战区消亡**：当前线被完全消灭，战区解散时，该战区所绑定的将领必须自动回到后备池（可用于下次派任）。
  - **新战区生成/分裂**：当因防线断裂产生了一个新战区（无初始将领），`GeneralDispatcher` 需要从同阵营的后备池中随机抽选一名可用将领赴任。若后备池无将领，则必须能安全降级（生成一个没有特定加成或最低属性的无名军官/保持空岗，取决于规则宽容度）。

### 3. LLM 换将接口准备
- **目标**：为之后大元帅（玩家角色或 LLM 总指挥）预留能够重新分配将领的途径。
- **实现方案**：在 `GeneralDispatcher` 内设计接口函数，例如 `func reassignGeneral(generalId: String, to zoneId: FrontZoneId) throws`。
- 当前不需要立刻对接大模型文本，只要保证如果通过代码强行调用该接口，游戏能在下一回合无缝完成换将，且 `TheaterCommanderPool` 在下一回合能根据最新的指派关系实例化正确的 `ZoneCommanderAgent`。

## 约束与验收标准
- **红线**：不改变 0.41 和 0.37 中建立的决策链路，只干预给战区匹配“谁”作为 `GeneralData`。
- **验收1（编辑器测试）**：能够通过自定义一份小地图 JSON，读取后自动生成带有两位具体历史将领的不同战区。
- **验收2（动态测试）**：在跑测试代码时，模拟战区因被切断导致分裂成2个 `Zone`。断言 `GeneralDispatcher` 自动捕获到此事件，并成功从后备池随机抽取了一个新将领填补空缺战区。模拟旧战区被摧毁后，原将领再次出现在后备池中可用。
- **验收3（接口测试）**：证明 `reassignGeneral` 可以安全移权。
