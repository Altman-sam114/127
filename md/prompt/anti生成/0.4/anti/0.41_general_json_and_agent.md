# v0.41 实体将领 JSON 数据接入与 ZoneCommanderAgent 适配

## 阶段目标
在 v0.3 中，我们已经完成了 `ZoneCommanderAgent` 和命令执行管线的统一收口，当前的战区会将自动生成的占位符（如 "German Commander"）挂载给战区指挥。
在 v0.41 中，我们需要正式引入历史将领实体（**不更改核心执行引擎**），并让具体的历史将领掌管战区。

## 需要实现的 6 位实体将领
请在数据层面实现以下六位将军：
- **盟军（Allies）**：蒙哥马利 (Montgomery)、魏刚 (Weygand)、戴高乐 (de Gaulle)
- **轴心国（Germany）**：古德里安 (Guderian)、里布 (Leeb)、博客 (Bock)

## 具体任务步骤

### 1. 建立将领数据模版 (JSON Layer)
在项目中创建类似于 `Data/generals.json` 的配置文件，定义以上 6 位将领。
字段必须能映射并在后续扩展，建议包含：
- `id`: 全局唯一（例如 `"general_guderian"`）
- `name`: 显示名称
- `faction`: 对应模型（`.germany` 或 `.allies`）
- `commandStyle`: `"aggressive" | "balanced" | "cautious"` (直接对应目前 `ZoneCommanderAgentConfig.CommandStyle`)
- `skills`: 特长标签（如 `"armor_expert"`, `"logistics"`, `"defensive_master"` 等）
- `portrait` / `biography`: UI展示需要的额外沉浸感描述字段（为0.43展示做准备）。

### 2. 建立 GeneralRegistry (模型解析层)
- 新增 `GeneralRegistry` 对象，用于在 App 启动或进入战局时，解析读取 `generals.json`。
- 新增基础层的数据结构 `GeneralData`，它有别于依附于战区的 `ZoneCommanderAgentConfig`。

### 3. 在 TheaterCommanderPool 中完成注入与替换
当前 `TheaterCommanderPool.automatic(for:)` 仍然是按照 `"auto_\(zone.id.rawValue)"` 和默认 `"German Commander"` 去 fallback 创建 `ZoneCommanderAgentConfig`。
- 修改或提供新的接口，允许传入具体的 `GeneralData`，将其转换为带有明确偏好和配置的 `ZoneCommanderAgentConfig` 并实例化 `ZoneCommanderAgent`。
- 注意：由于我们在 0.42 才会做“将领调度器与规则”，在 **0.41** 阶段，你可以写硬编码（hardcode）测试：如果生成了盟军战区，先直接从 Registry 拿蒙哥马利顶上；如果是德军新战区，拿古德里安顶上（或者写一个简易的一对一初始化映射），以保证打通从“JSON -> `ZoneCommanderAgent` -> 依据将领自己的 `CommandStyle` (如 aggressive) 发出防守/进攻指令”的全链路。

### 4. 保持下层执行完全隔离
- 原则：绝对不要为了实体将领而去修改 `WarCommandExecutor` 或底层的 `RuleEngine`。
- 执行器已经做到 issuer-agnostic，我们的改造**仅限于**替换下发指令的人，不改变具体行为引擎。

## 验收标准
1. `generals.json` 数据存在且包含指定六位将领，字段丰富。
2. `GeneralRegistry` 能够被测试单测覆盖，证明 JSON 解析成功。
3. `TheaterCommanderPool` 具备利用指定将领资源生成 `ZoneCommanderAgent` 的能力，并可在 log 或测试中断言生成指令的 issuerId 是具体的 `general_guderian` 而非通用的 `auto_zone`。
4. 跑通所有现有 `CommandSystemTests` 和 Regression Tests（不可损坏原有 v0.37 基线）。
