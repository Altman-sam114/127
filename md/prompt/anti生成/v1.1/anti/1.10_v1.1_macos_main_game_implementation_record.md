# v1.1 macOS 主游戏分支实现记录

日期：2026-07-04

分支：`v1.1-macos-main-game`

## 1. 目标

本分支目标是给主游戏增加 macOS 版本入口，使 `main` 系列主游戏能作为独立 macOS app target 存在，而不是只保留 iOS app 或 MapEditorMac。

本轮目标只覆盖：

- 新增主游戏 macOS app target：`WWIIHexV0Mac`。
- 复用现有主游戏 `AppContainer -> RootGameView -> BoardSceneView -> RuleEngine` 链路。
- 复用现有 `WWIIHexV0/Data/*.json` 默认游戏资源。
- 让主游戏 SpriteKit 棋盘在 macOS 下可用鼠标点击、拖拽平移、滚轮/触控板缩放。
- 保持玩家和 AI 命令仍经 `Command` / `ZoneDirective` / `WarCommandExecutor` / `RuleEngine`。
- 只做轻量检查，不跑 Xcode、不跑 XCTest、不启动模拟器或 app。

## 2. 非目标

- 不改战争规则、占领规则、补给规则、AI 决策算法或数据 schema。
- 不把 `MapEditorMac` 合并进主游戏 macOS target。
- 不做 Catalyst 目标；本分支选择原生 macOS app target。
- 不启动 macOS app 做烟测。
- 不跑 `xcodebuild build`、`xcodebuild test`、Probe、Smoke、Stage Regression、Full。
- 不回滚工作树里已经存在的 v0.7/v0.5/经济/统治者等未提交改动。

## 5. macOS target 设计

### 5.1 选择独立 macOS target

新增 target：

```text
WWIIHexV0Mac
```

构建设置：

- `SDKROOT = macosx`
- `SUPPORTED_PLATFORMS = macosx`
- `MACOSX_DEPLOYMENT_TARGET = 14.0`
- `PRODUCT_BUNDLE_IDENTIFIER = com.example.WWIIHexV0Mac`
- `MARKETING_VERSION = 1.1`
- `INFOPLIST_KEY_CFBundleDisplayName = "WWII Hex V1.1 Mac"`

选择独立 macOS target 的原因：

- iOS 主游戏 target 保持原样，不把它改成 Catalyst。
- `MapEditorMac` 仍是独立编辑器 target，不和主游戏 macOS app 混用入口。
- 主游戏 macOS app 可以有自己的 `@main`、窗口大小、菜单命令和 resource phase。

### 5.2 target source phase

`WWIIHexV0Mac` source phase 包含：

- 主游戏 App 层：`WWIIHexV0MacApp.swift`、`AppContainer.swift`
- Core / Commands / Rules / Agents / Turn
- UI 主游戏组件
- SpriteKit 主棋盘组件
- 当前工作树里已被源码引用的支持文件：经济、外交、统治者、将领 registry 等

`WWIIHexV0Mac` source phase 不包含：

- `Tests/*.swift`
- `Probes/*.swift`
- `MapEditor/*.swift`
- `MapEditorMacApp.swift`

### 5.3 target resource phase

`WWIIHexV0Mac` resource phase 包含与 iOS 主游戏相同的默认 JSON：

```text
ardennes_v0_scenario.json
ardennes_v02_regions.json
general_agents.json
generals.json
terrain_rules.json
unit_templates.json
```

DEBUG 下 `DataLoader` 仍会优先通过 `#filePath` 读取源码目录 `WWIIHexV0/Data/*.json`；bundle resource 是 release / fallback 路径。

## 6. macOS App 入口逻辑

新增 `WWIIHexV0MacApp.swift`：

```text
WWIIHexV0MacApp
  -> @StateObject AppContainer.bootstrap()
  -> WindowGroup
  -> RootGameView(container:)
  -> defaultSize 1440x900
  -> min frame 1200x760
```

新增 macOS 菜单：

- `Game / End Turn`：调用 `container.advanceOrRunAI`
- `Game / New Game`：调用 `container.resetGame`

这只是桌面快捷入口，不绕过规则系统。`End Turn` 仍落到 `AppContainer.advanceOrRunAI`，最终进入 `RuleEngine.execute(.endTurn)` 或 AI 回合链路。

## 7. 跨平台 SwiftUI 背景色

新增 `PlatformStyles.swift`：

```text
PlatformStyles.systemBackground
  macOS -> Color(nsColor: .windowBackgroundColor)
  iOS   -> Color(uiColor: .systemBackground)

PlatformStyles.tertiarySystemBackground
  macOS -> Color(nsColor: .underPageBackgroundColor)
  iOS   -> Color(uiColor: .tertiarySystemBackground)
```

替换主游戏 UI 中的 iOS-only 写法：

```text
Color(.systemBackground)
Color(.tertiarySystemBackground)
```

受影响 UI：

- `RootGameView`
- `HUDView`
- `CommandPanelView`
- `EventLogView`
- `UnitInspectorView`
- `RegionInspectorView`
- `AgentPanelView`

算法语义不变，只是把平台颜色解析从 UIKit 专属 API 改为条件编译适配。

## 8. SpriteKit macOS 输入桥接

### 8.1 BoardSceneView

`BoardSceneView` 改成平台分支：

```text
macOS:
  NSViewRepresentable
  -> BoardEventSKView: SKView
  -> BoardScene

iOS:
  UIViewRepresentable
  -> SKView
  -> BoardScene
```

SwiftUI 仍是状态源：

```text
RootGameView
  -> BoardSceneView(renderState:onHexTapped:)
  -> BoardScene.configure(renderState:onHexTapped:)
  -> AppContainer.handleBoardTap
```

AppKit 只负责 SKView 生命周期和输入事件，不保存游戏状态。

### 8.2 macOS 点击选择

新增 macOS 鼠标点击链路：

```text
mouseDown
  -> 记录 lastDragScenePosition
  -> totalDragDistance = 0

mouseDragged
  -> 累计拖拽距离
  -> 移动 camera

mouseUp
  -> 如果 totalDragDistance < tapThreshold
     -> event.location(in: scene)
     -> layout.pixelToHex
     -> state.map.contains
     -> onHexTapped(coord)
```

点击只调用现有 `onHexTapped`，后续仍由 `AppContainer.handleBoardTap` 判断选中、移动、攻击和提交命令。

### 8.3 macOS 拖拽平移

新增 macOS 拖拽平移：

```text
mouseDragged:
  current = event.location(in: scene)
  delta = current - previous
  camera.position.x -= delta.x
  camera.position.y -= delta.y
  clampCamera()
```

平移只改变视口 camera，不改 `GameState`。

### 8.4 macOS 滚轮/触控板缩放

`BoardEventSKView` 接收 AppKit 事件：

```text
scrollWheel
  -> view.convert(event.locationInWindow)
  -> scene.convertPoint(fromView:)
  -> BoardScene.handleScrollWheel(event, anchor:)

magnify
  -> BoardScene.handleMagnify(event, anchor:)
```

缩放逻辑：

```text
nextScale = clamp(oldScale * multiplier, 0.45...2.4)
ratio = nextScale / oldScale
camera.position = anchor + (camera.position - anchor) * ratio
camera.setScale(nextScale)
clampCamera()
```

该逻辑让缩放围绕鼠标所在 scene point，而不是固定窗口中心。

### 8.5 滚轮平移

滚轮/触控板水平滚动或按住 Shift 时：

```text
camera.position.x += scrollingDeltaX * camera.xScale
camera.position.y -= scrollingDeltaY * camera.yScale
clampCamera()
```

## 9. 与主游戏规则链路的关系

v1.1 不改变核心规则链路：

```text
玩家点击 macOS 棋盘
  -> BoardScene.onHexTapped
  -> AppContainer.handleBoardTap
  -> Command.move / Command.attack / selection
  -> RuleEngine
  -> CommandValidator
  -> CommandExecutor
  -> StrategicStateBootstrapper / synchronizer
  -> UI renderState 刷新
```

AI 链路也不因 macOS target 改变：

```text
End Turn / AI active
  -> AppContainer.advanceOrRunAI
  -> TurnManager
  -> WarCommandExecutor
  -> RuleEngine
```

因此本分支是平台承载和输入适配，不是规则/AI 算法升级。

- 未跑 Xcode 构建，`WWIIHexV0Mac` target 是否能完整编译仍未由构建验证确认。

- `project.pbxproj` 手工加入大量 source phase 条目，已通过 `plutil`，但未通过 Xcode 工程加载验证。

建议下一步：

- 人工授权后再跑 `xcodebuild -scheme WWIIHexV0Mac build` 或在 Xcode 中打开 target 做真实编译。
- 若要隔离 v1.1，可在干净 worktree 中只挑选 macOS target、平台桥接和文档改动，避免混入 v0.7/v0.5 脏改动。
