# v0.5 大版本：决策链跃升与大模型 (LLM) 真正降临

## 1. v0.5 版本定位与雄心
在 v0.3 中，我们构建了可靠的战争执行引擎（战区与前线）；在 v0.4 中，我们实体化了将军并完善了玩家强行干预的双轨 UI 制。
现在，**v0.5 的终极目标是将“灵魂”注入这套系统**。我们要将指挥层级拔高到“大战略级别”（元帅），并彻底规范决策指令链，为真正调用本地或云端 LLM 铺设完整的 JSON 的生成与解码管道（IO 接口）。

这是一个**脱胎换骨**的里程碑，它标志着游戏从“死板规则战棋”向“活体 AI 推演大局”的全面转型。

---

## 2. 核心架构与挑战分析

### 2.1 决策链规范化 (Decision Chain Normalization)
在 v0.4 中，命令流是：`玩家 / GeneralAgent -> ZoneDirective -> WarCommandExecutor`。
在 v0.5 中，我们需要加入**元帅层 (Marshal / Theater Group Commander)**，彻底确立**三级垂直指挥链**：

`[未来统治者 (Ruler)] -> [元帅 (Marshal)] -> [战区将军 (General)] -> [战术执行引擎 (Executor)]`

*   **元帅的定位**：元帅不点对点管理“装甲师”，他管理的是**“多个战区” (FrontZones)** 和**“战略预备队”**。
*   **指令层级化**：
    *   元帅下达 **`TheaterDirective`** 比如：“优先把大部分后勤和预备队向北倾斜，北线要求强攻，南线给我拖住”。
    *   战区将军接收这个 `TheaterDirective`，结合自身所在战区的局部局势，将其解释为具体的 **`ZoneDirective`** 报给执行器。

### 2.2 大模型输入：战场状态摘要生成器 (JSON State Summarizer)
**挑战**：把整个地图上千个 Hex，几十上百个师的数据直接丢给 LLM，会导致 Context Window 爆炸而且绝对会发生大模型幻觉瞎指挥。
**解决核心**：**绝对不要给模型看底层网格**。模型只需要看抽象降维后的 JSON。
*   **需要开发的组件**：`ContextSummarizer`。
*   **数据抽象**：
    ```json
    {
      "marshal_name": "伦德施泰特",
      "personality": "均衡、稳重",
      "overall_supply": "充足",
      "fronts": [
        {
          "zone_id": "arden_north",
          "general": "古德里安",
          "strength_ratio": "1.5",
          "key_objectives_held": ["列日"],
          "key_objectives_lost": [],
          "status": "交火剧烈"
        }
      ]
    }
    ```
    我们要通过算法，把战场态势“翻译”成上面的情报简报，喂给 LLM。

### 2.3 大模型输出：结构化解码与容错拦截 (LLM Decoder & Fallback)
**挑战**：LLM 哪怕再聪明，也可能会乱吐 Markdown 以及不存在的 `zone_id`。
**解决核心**：
*   要求 Prompt 里强制包含 ````json` 格式要求。
*   开发在服务层的 `LLMDecoder` 组件：
    *   **提取器**：用正则提取回答中 ````json` 到 ` ```` ` 之间的内容。
    *   **解码器**：通过 Swift 原生 `JSONDecoder` 尝试反序列化成 `TheaterDirective` 结构体。
    *   **容错管线**：如果在解码或 ID 映射时失败，要有重试逻辑 (Retry Mechanism) 或者在 2 次失败后，自动降级为走老 AI（传统的掷骰子决策树）。

## 3. v0.5 落地细分步调建议 (Sub-version Breakdown)

为了保证能够安全、平滑地接轨 LLM 且系统不崩，强烈建议按以下 0.51 - 0.55 的顺序逐个击破：

### ✅ **v0.51 元帅实体化与多前线编组**
像 0.41 引入将军那样，在 JSON 中引入几位元帅（如：伦德施泰特、朱可夫）。
扩展数据结构，使一个元帅拥有一个 `TheaterGroup` 属性，能够辖制 2-3 个 `FrontZone` 生成级联管理数组。

### ✅ **v0.52 垂直决策链建立（白盒测试阶段）**
完成 `MarshalAgent -> TheaterDirective -> GeneralAgent -> ZoneDirective` 逻辑转换。
第一步先**不接 LLM**，而是写一组简单的硬编码逻辑：当最高指令要求“左翼突破”时，左边战区的将军触发激进算法，右边战区退防。并提供明确的接口签名，等待外部大脑注入。

### ✅ **v0.53 情报官系统：无缝战场降维收集 (The Summarizer)**
开发核心引擎组件 `BattlefieldSummarizer`。
它可以把 `GameState`, `MapState`, 资源情况，压缩成符合 LLM Token 上限阈值的情报结构。可以在测试环境下直接输出成一段文本，供开发人员进行人工审阅“这情报我能不能看懂”。

### ✅ **v0.54 外交官系统：LLM JSON 强制输出与硬核反序列化 (The Decoder)**
搭建大模型通信客户端。设定死板的 Few-Shot Prompt，规范大模型只能吐带有明确 Schema 的可执行 JSON。开发强悍的反序列化与结构体拦截器，验证各种错乱输出情况下的丢弃机制正常运行。

### ✅ **v0.55 终极合笼：第一声炮响**
正式让伦德施泰特接管某个多战区。回合初调用你本地的大模型 API（如 Ollama 挂载的本地端或是通过网络端接口），接收情报，沉思几秒后，打印出他发往各战区的 JSON 指令。系统接收 -> 转译 -> 部队出击。

## 4. 架构安全警告 (Red Alerts)
1. **幻觉越权**：切记在 Decoder 里过滤所有不合法的 Region 操作。LLM 没有权限执行“让第七装甲师瞬移到伦敦”，它生成的战役意图只能告诉引擎“我要打伦敦方向”，如果由于无路可走失败，那是引擎的问题，不能因 LLM 的乱写使游戏 crash。
2. **延迟与异步**：大模型思考可能需要几秒钟甚至十几秒。原本同步的 `TurnManager.runAITurn()` 必须被重构为**基于现代协程 (`async / await`) 的异步操作流**，并在画面上给玩家打出一个“正在进行军械部研判 (Generals are Planning...)”的等待 UI。
