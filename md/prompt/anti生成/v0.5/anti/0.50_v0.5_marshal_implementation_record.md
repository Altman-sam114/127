# v0.5 元帅层与模拟 LLM JSON 管线实现记录

目标分支：`v0.5-marshal-decision-chain`

分支审计说明：本轮开始时已创建并切换过 `v0.5-marshal-decision-chain`。后续当前 checkout 多次被外部/并行上下文推进到其他版本分支，审计中先后观察到 `v0.9-ruler-diplomacy`、`v0.4-generals-command-ui-resume`、`v1.1-macos-main-game`、`v1.0-ui-ai-playtest` 等非 v0.5 分支名，且工作树内存在多批其他版本未提交改动。用户同意切换后，当前 checkout 已确认回到 `v0.5-marshal-decision-chain`；合并前仍需审查 dirty worktree 中非 v0.5 文件归属和冲突风险。

本记录说明当前分支真实改动。v0.5 的目标是把默认战争 AI 从“每个 FrontZone 的将军直接出 `ZoneDirective`”提升为“元帅先读降维战场摘要，输出元帅级 `TheaterDirective` JSON，再编译成将军/战区级 `ZoneDirective`”。真实执行仍然只走 `WarCommandExecutor -> RuleEngine`。

## 1. 本分支完成内容

### 1.1 元帅层引入

新增运行时类型：

- `MarshalAgentConfig`
  - 字段：`id`、`name`、`faction`、`personality`、`strategicBias`、`theaterGroupZoneIds`。
  - 作用：定义元帅身份和辖制的 FrontZone 组。
  - 当前默认：
    - 德军：`marshal_rundstedt`，偏进攻。
    - 盟军：`marshal_eisenhower`，偏均衡。

- `MarshalAgent`
  - 作用：回合级元帅决策入口。
  - 输入：`GameState`、当前阵营、fallback `TheaterCommanderPool`。
  - 输出：`MarshalDirectiveResolution`，包含原始模拟 LLM JSON、解码后的 `TheaterDirectiveEnvelope`、编译后的 `DirectiveEnvelope`、诊断信息。

元帅不是单位，也不是地图实体，不直接管理 hex、region controller、单位坐标或 `GameState`。元帅只发“战役意图”。

### 1.2 决策链规范化

v0.5 默认链路：

```text
TurnManager.runAITurn(.marshalDirective)
  -> MarshalBattlefieldSummarizer.summary
  -> SimulatedMarshalLLMClient.completeTheaterDirectiveJSON
  -> TheaterDirectiveDecoder.parse
  -> TheaterDirectiveCompiler.compile
  -> DirectiveEnvelope / ZoneDirective
  -> WarCommandExecutor.execute
  -> RuleEngine.execute(Command)
  -> WarDirectiveRecord / AgentDecisionRecord
```

旧 v0.37 链路仍可显式使用：

```text
TurnManager.runAITurn(.zoneDirective)
  -> TheaterCommanderPool
  -> ZoneCommanderAgent
  -> ZoneDirective
  -> WarCommandExecutor
```

Legacy Agent D 仍只在 `.legacyAgentOrder` 显式分支使用，未接回默认战争 AI 主路径。

### 1.3 JSON 输入输出与 decoder

新增 LLM-facing schema：

- `TheaterDirectiveEnvelope`
  - `schemaVersion = 5`
  - `issuerId`
  - `turn`
  - `faction`
  - `strategicIntent`
  - `directives: [TheaterDirective]`
  - `summary`

- `TheaterDirective`
  - `zoneId`
  - `category`
  - `tactic`
  - `priority`
  - `targetTheaterId`
  - `weightedRegions`
  - `focusRegionId`
  - `supportRegionIds`
  - `reserveBias`
  - `intensity`
  - `maxCommittedUnits`
  - `exploitDepth`
  - `rationale`

新增 `TheaterDirectiveDecoder`：

1. 从原始文本中优先提取 fenced code block：先找 ````json`，再找普通 ```。
2. 使用 `JSONDecoder` 解码 `TheaterDirectiveEnvelope`。
3. 校验：
   - schemaVersion 必须在支持集合内，当前是 5。
   - issuerId / turn / faction 必须匹配预期。
   - 每条 directive 的 `zoneId` 必须存在。
   - zone 阵营必须等于 envelope 阵营。
   - tactic 的 category 必须与 directive.category 一致。
   - target theater 或 fallback front zone 必须存在。
   - 所有 region id 必须存在。
4. 失败时不执行半成品 JSON，交给 `MarshalAgent` fallback 到 `TheaterCommanderPool`。

decoder 不把 LLM 输出转成 `Command`，只接受元帅级意图，再由 compiler 降级成现有 `ZoneDirective`。

### 1.4 模拟 LLM 接入

新增 `MarshalLLMClient` 协议与 `SimulatedMarshalLLMClient`。

当前不接真实网络或本地模型。模拟客户端会：

1. 接收 `MarshalBattlefieldSummary`。
2. 按 front strength ratio、供应警告、战略偏好生成 `TheaterDirectiveEnvelope`。
3. 用 `JSONEncoder` 输出 pretty JSON。
4. 故意包在 ````json` fenced block 中，覆盖 decoder 的提取路径。

这满足“LLM 接入测试”的接口形态，但没有外部 API、端口、网络调用或模型依赖。

### 1.5 战场摘要算法

新增 `MarshalBattlefieldSummarizer`，只给元帅看降维摘要，不给全量 hex 网格。

摘要字段：

- 回合、阵营、元帅身份、人格、战略偏好。
- 总体补给：`adequate` / `strained` / `encircled_risk`。
- 我方低补给和包围单位数量。
- 已守住/丢失目标名称。
- 每个 front zone 的 `MarshalFrontSummary`：
  - zone id/name/state/pressure。
  - front region ids、enemy region ids、enemy zone ids。
  - friendly front strength、friendly depth strength、visible enemy strength。
  - strength ratio。
  - front/depth/garrison 单位数量。
  - supply warning 数量。
  - 关键目标 held/lost。
  - status：`supply_warning` / `under_pressure` / `advantage` / `outnumbered` / `stable_contact`。
- 最近事件，默认取最后 8 条。

摘要只读取 region、front zone、deployment、unit strength、objective 和 event log，不把每个 hex tile 列给模型。

### 1.6 TheaterDirective -> ZoneDirective 编译算法

新增 `TheaterDirectiveCompiler`。

输入：元帅 JSON 解码后的 `TheaterDirectiveEnvelope`、当前 `GameState`、fallback `TheaterCommanderPool`。

编译规则：

- 对当前阵营每个有前线的 `FrontZone` 处理。
- 如果该 zone 有元帅 directive：
  - `category == offense`：
    - 生成 `ZoneDirective.attack`。
    - target theater 优先用元帅给的 `targetTheaterId`，否则取该 zone 第一个 enemy zone。
    - weighted regions = `focusRegionId + weightedRegions + supportRegionIds` 去重。
    - 保留元帅给的 `intensity`、`maxCommittedUnits`、`exploitDepth`。
    - `commandTarget` 优先 region，否则 theater。
  - `category == defense`：
    - 生成 `ZoneDirective.defend`。
    - `targetReserves = max(1, reserveBias)`。
    - `strongpointRegionIds = focusRegionId + weightedRegions`。
    - `fallbackRegionIds = supportRegionIds`。
    - `maxFrontCommitment = maxCommittedUnits`。
- 如果该 zone 没有元帅 directive 或编译失败，使用 fallback `TheaterCommanderPool` 为该 zone 生成的旧 `ZoneDirective`。

编译器不执行命令、不修改状态，只产出 `DirectiveEnvelope`。

### 1.7 TurnManager 接入

`WarPipelineMode` 新增：

```swift
case marshalDirective
```

`TurnManager` 新增 `marshalAgent` 可选依赖，默认 pipeline 改为 `.marshalDirective`。

执行逻辑抽为公共 `executeDirectiveEnvelope`：

- `.marshalDirective` 先解析元帅 JSON，再编译成 `DirectiveEnvelope`。
- `.zoneDirective` 仍直接由 `TheaterCommanderPool` 生成 `DirectiveEnvelope`。
- 两者之后共用同一个执行路径：
  - 对每条 `ZoneDirective` 调 `WarCommandExecutor.execute`。
  - 写 `WarDirectiveRecord`。
  - 自动执行 `.endTurn`。
  - 写 `AgentDecisionRecord`。

因此元帅层不会分叉出第二套执行器。

本次收口还移除了 v0.9 旁支曾插入的 `RulerAgent` 塑形调用：`.marshalDirective` 与显式 `.zoneDirective` 当前都直接进入 `executeDirectiveEnvelope`，不写 `RulerDecisionRecord`，不在 v0.5 默认链路中接入统治者层。

### 1.8 AppContainer 接线

`AppContainer` 默认 `warPipelineMode` 改为 `.marshalDirective`。

启动和观察者模式中按阵营自动构造元帅：

```text
AppContainer.bootstrap
  -> buildCommanderPool
  -> buildMarshalAgent(.germany)
  -> TurnManager(... marshalAgent:)

observer allies AI
  -> buildMarshalAgent(.allies)
```

## 2. 本分支没有做的事

- 没有接真实 LLM、本地模型、网络 API、Ollama 或云端接口。
- 没有恢复旧 v0.5 `CabinetState` / `DirectiveBoard` / `MinisterDecisionProvider` / `RulerDirectiveFactory` / `national_cabinet.json`。
- 没有把 `GameState` 扩展为国家内阁或部长系统。
- 没有把统治者层接入 v0.5 当前主链路。
- 没有让 LLM 或元帅直接输出 `Command`。
- 没有让元帅直接修改 hex controller、region controller、unit coord、`hexToTheater` 或 `hexToFrontZone`。
- 没有删除 Legacy Agent D。
- 没有跑 Xcode / XCTest / 模拟器 / Probe / Smoke / Full。

## 3. 与 v0.7/v0.9 工作树改动的关系

当前工作树已经存在未提交的高级战术、外交、经济、UI 和地图编辑器方向改动，例如：

- `TacticName` 扩展到 `blitzkrieg`、`breakthrough`、`fireCoverage` 等。
- `AttackParameters` / `DefenseParameters` 增加 focus/support/convergence/maxCommitment 等字段。
- `WarCommandExecutor` 有高级战术执行策略。
- 工作树存在未跟踪 `WWIIHexV0/Core/DiplomacyState.swift`、`WWIIHexV0/Core/EconomyState.swift`、`WWIIHexV0/Agents/RulerAgent.swift`、`WWIIHexV0/Data/generals.json` 等其他方向文件；其中 `RulerAgent` 不在 v0.5 `TurnManager` 主链路中调用。

这些改动不是本 v0.5 元帅层的核心要求。本轮未回滚它们，而是在现有工作树基础上兼容：

- 元帅 JSON 可以携带 `maxCommittedUnits` / `exploitDepth` / `supportRegionIds`，compiler 会传给现有 `AttackParameters` / `DefenseParameters`。
- 元帅链路仍以 `ZoneDirective` 为下游边界，不依赖外交状态。
- 合并前必须由人工或后续 Agent 重新确认目标分支、文件归属和版本边界；不能仅凭当前 dirty worktree 直接合并。
