# v6.0 现代战争迁移审计与兼容合同

> 本记录对应 `codex-v6.0-现代战争aiagent迁移总提示词.md` 的第一阶段落地。当前目标是建立迁移合同、审计清单和低风险显示层兼容，不代表 v6.1-v6.10 的规则、剧本、ISR/EW/火力和发布级 UI 已完成。

## 1. 本轮目标

- 明确现代战争产品目标：`Modern Command Agent` / `现代联合指挥 Agent`。
- 保持 Hex / Region / Theater / FrontLine / WarDeployment 分层不变。
- 保持玩家、AI、MockAI、后续聊天命令最终收口到 `Command` / `ZoneDirective -> WarCommandExecutor -> RuleEngine`。
- 先清理主 UI 的玩家可见二战显示名，不立即强改 `Faction` raw value、`GamePhase` raw value、JSON schema 或规则执行。
- 建立 v6.1 之前必须处理的二战硬编码、二元敌我和中立 fallback 审计表。

## 2. 非目标

- 不把 `Faction.germany/allies` 一次性重命名为 `blueForce/redForce`。
- 不删除 `Faction.opponent`，只记录为 v6.1 的最高风险迁移点。
- 不切换默认剧本到 `grey_tide_2030`，新剧本应在 v6.2 单独落地。
- 不新增 ISR、ContactTrack、EW、FireMission、AirTasking 或 LogisticsNetwork 运行时状态。
- 不改 `WWIIHexV0.xcodeproj/project.pbxproj`，避免项目文件引用冲突。
- 不运行本机 Xcode build、XCTest、模拟器、Probe、Smoke 或 Full。

## 3. 已落地的兼容显示层

本轮采用“raw value 保持旧名，显示名转现代语义”的兼容策略。这样旧 JSON、旧测试 fixture、旧存档仍能解码，玩家主要面板开始脱离 Germany / Allies / Panzer / Ardennes V0。

| 类型/位置 | 兼容保留 | 现代显示 |
|---|---|---|
| `Faction.germany` | raw value 仍为 `germany` | `Red Operational Group` / `Red Force` |
| `Faction.allies` | raw value 仍为 `allies` | `Blue Joint Task Force` / `Blue Force` |
| `GamePhase.germanAI` | raw value 仍为 `germanAI` | `AI Command` |
| `GamePhase.alliedPlayer` | raw value 仍为 `alliedPlayer` | `Player Command` |
| `ProductionKind.panzerDivision` | raw value 仍为 `panzerDivision` | `Armored Task Force` |
| `ProductionKind.motorizedDivision` | raw value 仍为 `motorizedDivision` | `Mechanized Task Force` |
| `MapDisplayLayer.province` | raw value 仍为 `province` | `Sector` |
| `MapDisplayLayer.initialTheater` | raw value 仍为 `initialTheater` | `Baseline` |
| `MapDisplayLayer.dynamicTheater` | raw value 仍为 `dynamicTheater` | `Operational` |
| `MapDisplayLayer.frontLine` | raw value 仍为 `frontLine` | `Contact` |
| `MapDisplayLayer.deployment` | raw value 仍为 `deployment` | `Brigade` |
| `Division.name` | 编码字段仍保存原数据名 | UI 读取 `operationalDisplayName` |

`Division.operationalDisplayName` 只做玩家显示映射，不改变持久化字段。例如：

- `1st Panzer Division` -> `1st Armored Task Force`
- `2nd Motorized Division` -> `2nd Mechanized Task Force`
- `101st Infantry Division` -> `101st Infantry Task Force`
- `9th Anti-Tank Battalion` -> `9th Anti-Armor Team`
- `Bastogne Garrison` -> `Objective Security Detachment`

## 4. 审计清单

### 4.1 二元作战方与敌我关系

当前状态：

- `WWIIHexV0/Core/Faction.swift` 仍只有 `.germany` / `.allies`。
- `Faction.opponent` 仍被 `WarCommandExecutor`、`SupplyRules`、`AgentContexts`、`RegionCombatRules`、`RegionSupplyRules`、`FrontLineManager`、`ZoneCommanderAgent` 等使用。
- `DiplomacyState` 已存在 country / bloc / relation 草案，但底层敌我判断尚未收口到 ROE helper。

v6.1 要求：

- 增加或桥接 `blueForce`、`redForce`、`neutral`。
- 新增敌我 helper，例如 `isHostile(_:_:)` / `rulesOfEngagement(for:against:)`。
- 逐步替换 `Faction.opponent` 调用，不允许多方现代战争继续依赖二元 opponent。

### 4.2 回合阶段与控制权

当前状态：

- `GamePhase.germanAI/alliedPlayer` 仍参与 `CommandExecutor`、`CommandValidator`、`TurnManager`、`AppContainer` 的 active side 判断。
- 本轮只改 `displayName` 为 `AI Command` / `Player Command`。

v6.1 要求：

- 保留旧 raw value 兼容时，必须新增通用 phase 语义或控制 helper。
- `AppContainer.shouldRunAI` 应基于 side control profile，而不是硬编码 Germany / Allies。
- `CommandValidator` 的玩家/AI 校验要对称，不因显示名现代化而改变规则。

### 4.3 默认数据与 JSON

当前状态：

- `DataLoader.loadInitialGameState()` 默认仍加载 `ardennes_v0_scenario` 和 `ardennes_v02_regions`。
- `DataLoader` fallback components 仍包含 `tank_division`、`panzer_division`、`motorized_division`。
- `general_agents.json`、`generals.json`、`unit_templates.json` 和阿登 JSON 仍是二战数据。
- `RegionDataSet.toRegions()` 对 nil owner/controller 仍 fallback 到 `.allies`。

v6.2 要求：

- 新增 `grey_tide_2030_scenario.json`、`grey_tide_2030_regions.json`、`modern_unit_templates.json`、`modern_commanders.json`。
- 默认新局切换到现代剧本。
- 中立 owner/controller 不能再 fallback 到 `.allies`。

### 4.4 UI / SpriteKit 玩家可见层

本轮已改：

- `HUDView` 主标题、Side、Personnel、Materiel、Sustainment。
- `RootGameView` 主地图 accessibility label 和 tabs：Formation、Sector、Command、Sustainment、ROE。
- `EconomyPanelView` 显示为 Sustainment / Force Packages。
- `UnitInspectorView` 显示 Formation / Side / Operational Zone / Brigade Sector。
- `UnitTooltipView` 显示 `operationalDisplayName`。
- `BoardScene` 空状态标题改为 `Modern Command Board`。

仍需处理：

- Agent 面板和日志中可能仍展示旧 JSON / 旧单位原始名。
- MapEditor UI 仍有摩托化师、germany/allies、panzer/tank 判断。
- 默认数据仍会通过 raw JSON 带入旧地名和旧指挥官名。

### 4.5 MapEditor

当前状态：

- `MapEditorGameResourceBridge` 默认资源仍是阿登。
- `MapEditorExporter` 导出默认 `alliedPlayer`、`allies`、`germany`。
- `MapEditorView` 和 `MapEditorViewModel` 仍使用摩托化师、ger/all prefix、germany/allies 选择。

v6.2 要求：

- 编辑器术语迁移为地块、作战区、任务区、部队/装备/任务编组。
- 支持现代地形和关键节点显示名。
- 保持导出 schema 与主游戏 `ScenarioDefinition` / `RegionDataSet` 兼容。

## 5. 下一阶段接口合同

v6.1 不应直接大规模重命名全项目。建议顺序：

1. 在 `Faction` 或新兼容类型中引入现代 side 语义，保留旧 raw value decode fallback。
2. 在 `DiplomacyState` 周围增加 ROE / relation helper。
3. 把 `Faction.opponent` 的使用点按规则风险从高到低替换：命令校验、补给、前线、AI summary、WarCommandExecutor。
4. 修 `RegionDataSet.toRegions()` 的 nil fallback，明确 neutral 表达。
5. 再切默认数据和 MapEditor 术语。

## 6. 轻量检查边界

本轮允许：

- `git diff --check`
- 文档尾随空白扫描
- 冲突标记扫描
- 少量纯 Swift 文件的 `swiftc -parse`，如果跨文件依赖导致不可行则记录风险

本轮禁止：

- 本机 `xcodebuild build`
- 本机 `xcodebuild test`
- Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full
- 模拟器或 app 启动

## 7. 风险

- 显示层现代化不等于规则现代化；底层仍是二元 Germany / Allies。
- 默认阿登 JSON 仍会进入事件日志、AI raw JSON、测试 fixture 和 MapEditor。
- 未运行本机 build，SwiftUI / SpriteKit 编译正确性需要等待云端或人工授权重验证。
- v6.1 替换 `Faction.opponent` 时必须小步推进，否则容易破坏补给、前线、部署和 AI 指令管线。
